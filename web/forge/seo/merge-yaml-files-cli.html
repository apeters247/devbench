Merge YAML Files from the Command Line


Merging two YAML files sounds easy until you hit lists: should containers: [app, sidecar] in the overlay replace the base list, extend it, or partially override individual items? devbench cf --merge lets you choose explicitly.


devbench cf base.yaml --merge overlay.yaml
devbench cf base.yaml --merge overlay.yaml --list-merge append
devbench cf base.yaml --merge overlay.yaml --list-merge merge

Three List Strategies


StrategyWhat happens to listsBest for
replace (default)Overlay list replaces base list entirelySimple config overrides, feature flags
appendOverlay list appended to base listAdding env vars, volumes, firewall rules
mergePositional deep-merge (overlay[0] merges into base[0])Patching Kubernetes containers without repeating every field

Strategy 1: replace (default)


Dict keys are deep-merged; lists are replaced wholesale. This is the safe default when the overlay should win completely.


# base.yaml
database:
  host: localhost
  port: 5432
  replicas: 2


# overlay.yaml
database:
  host: db.prod.example.com


devbench cf base.yaml --merge overlay.yaml

database:
  host: db.prod.example.com
  port: 5432
  replicas: 2

Nested dict keys from the overlay override base values; keys present only in base are kept.


Strategy 2: append


The overlay list is concatenated onto the base list. Useful for adding Kubernetes env vars or volumes without duplicating the whole base list in your overlay file.


# base.yaml
env:
  - name: APP_ENV
    value: production
  - name: LOG_LEVEL
    value: info


# overlay.yaml
env:
  - name: DEBUG
    value: "true"
  - name: FEATURE_FLAG_X
    value: "1"


devbench cf base.yaml --merge overlay.yaml --list-merge append

env:
  - name: APP_ENV
    value: production
  - name: LOG_LEVEL
    value: info
  - name: DEBUG
    value: 'true'
  - name: FEATURE_FLAG_X
    value: '1'

Strategy 3: merge (positional deep-merge)


Each overlay list item is deep-merged into the corresponding base item by position. Items in the overlay beyond the base list length are appended. This solves the yq pain point where you want to update just the image of a container without repeating every other field.


# base.yaml
containers:
  - name: app
    image: myapp:v1
    env:
      - name: APP_ENV
        value: production
    resources:
      limits:
        cpu: "500m"
        memory: 512Mi
  - name: sidecar
    image: envoy:1.25


# overlay.yaml — only update the app image
containers:
  - image: myapp:v2


devbench cf base.yaml --merge overlay.yaml --list-merge merge

containers:
  - name: app
    image: myapp:v2        # updated
    env:
      - name: APP_ENV
        value: production  # preserved
    resources:
      limits:
        cpu: 500m
        memory: 512Mi      # preserved
  - name: sidecar
    image: envoy:1.25      # unchanged

The merge strategy works recursively — nested dicts inside list items are also deep-merged, so you can patch a single key deep in a list element without repeating the entire element.


Cross-Format Merging


The base and overlay can be in different formats. devbench auto-detects both:


# Merge a TOML overlay onto a YAML base; output stays YAML
devbench cf base.yaml --merge overlay.toml

# Merge JSON overlay onto TOML base; output stays TOML
devbench cf base.toml --merge overlay.json

# Override format explicitly
devbench cf base.yaml --merge overlay.json --to yaml

Merging Into CI/CD Pipelines


# Patch a Kubernetes deployment without modifying the base file
devbench cf k8s/base/deployment.yaml \
  --merge k8s/overlays/production/deployment.yaml \
  --list-merge merge \
  | kubectl apply -f -

# Override dev config with secrets from env
devbench cf config.dev.yaml --merge config.secrets.yaml > config.merged.yaml


# GitHub Actions example
- name: Merge environment-specific config
  run: |
    pip install devbench
    devbench cf config.base.yaml \
      --merge config.${{ matrix.env }}.yaml \
      --list-merge append \
      -t json > config.json

Inspect Without Merging


# Diff two YAML files before merging
devbench cf base.yaml --diff overlay.yaml

# Structural diff as JSON (CI-friendly)
devbench cf base.yaml --diff overlay.yaml --raw

Install


pip install devbench
devbench cf --help

Works on Python 3.10–3.13. Fully offline — no telemetry, no external API calls. Supports 11 formats: JSON, YAML, TOML, XML, CSV, INI, .env, HCL, Java .properties, Apple plist, JSONC.
