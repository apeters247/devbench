Validate Config Files in CI/CD Pipelines

One tool for YAML, JSON, TOML, INI, .env, and 6 more — with CI-native exit codes

The Problem: Config Errors Hit Production

A misplaced indent in a Kubernetes YAML, a trailing comma in a JSON config, a missing section header in an INI file — these break deployments. Most teams only catch them when the deploy fails. A proper pre-commit hook or CI step catches them in seconds.

The challenge: your configs span multiple formats. Kubernetes uses YAML. Terraform uses HCL. Node.js uses JSON. Python apps use INI or .env. You'd need four separate linters. Or one ConfigForge.

ConfigForge --validate

ConfigForge's --validate flag checks whether a config file is parseable. It exits 0 if valid, exits 1 if not — exactly what shell scripts and CI systems expect.

Single file

devbench cf config.yaml --validate
# config.yaml: valid  (yaml, 15 keys)

devbench cf broken.yaml --validate
# broken.yaml: INVALID
#   error: mapping values are not allowed here (line 4, col 5)

Batch validation — validate every config file at once

# Validate all YAML files in the current directory
devbench cf "*.yaml" --batch --validate

# Recursive — validate across subdirectories
devbench cf "**/*.yaml" --batch --validate -R

# Output:
# k8s/deployment.yaml: valid  (yaml, 12 keys)
# k8s/service.yaml: valid  (yaml, 6 keys)
# k8s/broken.yaml: INVALID
#   error: could not find expected ':' (line 8, col 3)
# [validate] 3 files: 2 valid, 1 invalid

GitHub Actions — Validate All Configs Before Merge

# .github/workflows/config-lint.yml
name: Validate Configs

on: [pull_request]

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install devbench
      - name: Validate YAML configs
        run: devbench cf "**/*.yaml" --batch --validate -R
      - name: Validate TOML configs
        run: devbench cf "**/*.toml" --batch --validate -R
      - name: Validate JSON configs
        run: devbench cf "**/*.json" --batch --validate -R

Each step exits 1 if any file is invalid, failing the PR check with a clear error message. Total install time: under 2 seconds on a cold runner.

pre-commit Hook

# .pre-commit-config.yaml
repos:
  - repo: local
    hooks:
      - id: validate-yaml
        name: Validate YAML configs
        language: system
        entry: devbench cf --validate --batch
        args: ["**/*.yaml", "-R"]
        pass_filenames: false

      - id: validate-toml
        name: Validate TOML configs
        language: system
        entry: devbench cf --validate --batch
        args: ["**/*.toml", "-R"]
        pass_filenames: false

Count Config Keys — Shell Scripting with ConfigForge

Beyond validation, --count PATH makes shell scripting with configs ergonomic: 

# Count replicas in a Kubernetes deployment
REPLICAS=$(devbench cf deployment.yaml --count spec.replicas)
echo "Deploying $REPLICAS replicas"

# Count containers
CONTAINERS=$(devbench cf deployment.yaml --count spec.template.spec.containers)
if [ "$CONTAINERS" -gt 5 ]; then
  echo "Warning: $CONTAINERS containers in one pod"
fi

# Count environment variables
ENV_VARS=$(devbench cf deploy.yaml --count spec.template.spec.containers[0].env)
echo "Pod has $ENV_VARS environment variables"

--count outputs a plain integer — no jq, no Python, no awk. Works with any supported format: YAML, JSON, TOML, INI, HCL, and more.
