ConfigForge vs Miller (mlr): Different Tools for Different Jobs


If you're searching for a Miller (mlr) alternative that handles YAML, TOML, and other config formats, ConfigForge (the cf subcommand in the devbench toolkit) covers formats that Miller doesn't support. This is an honest comparison of what each tool does well.


The Short Version


  • Use Miller when you need stream-processing, aggregation, or transformation of large CSV/TSV/JSON-lines datasets — Miller is purpose-built for this and extremely fast.
  • Use ConfigForge when you need to convert between YAML, TOML, HCL, INI, .env, plist, JSONC, and other config formats — Miller simply doesn't support these.
  • Use both in pipelines — ConfigForge converts the format, Miller processes the records.

Where Miller Wins


Miller is a data processing powerhouse for structured tabular data. Its DSL (Miller Record Language) supports:


  • Field transforms: mlr put '$price = $price * 1.1'
  • Aggregation: mlr stats1 -a mean,stddev -f price -g category
  • Sorting and filtering: mlr filter '$quantity > 100'
  • Join operations: mlr join -f lookup.csv -j id
  • Streaming billion-row files: constant memory, doesn't load the full file

For data engineering pipelines over CSV, TSV, and JSON Lines, Miller is faster and more expressive than ConfigForge. ConfigForge doesn't try to replicate this.


Where ConfigForge Wins


ConfigForge handles the entire config-file ecosystem that Miller doesn't touch:


# Convert Terraform HCL vars to YAML for Ansible
devbench cf terraform.tfvars.hcl -t yaml

# Convert Kubernetes YAML to JSON for tooling
devbench cf deployment.yaml -t json

# Extract database config from YAML as .env
devbench cf config.yaml --get database --to env > .env.db

# Convert Spring Boot properties to TOML for a Rust service
devbench cf application.properties -t toml

# Batch convert all TOML files in a project to YAML
devbench cf "**/*.toml" --batch -t yaml

# Validate pyproject.toml against schema
devbench cf pyproject.toml --validate

Comment preservation is ConfigForge's differentiator. When you convert a TOML file with operational comments to YAML, the comments survive:


# Production — rotate quarterly
[database]
host = "db.internal"
pool_size = 10  # increase if p99 > 200ms


devbench cf config.toml -t yaml

# Production — rotate quarterly
database:
  host: db.internal
  pool_size: 10  # increase if p99 > 200ms

Using Both in a Pipeline


ConfigForge and Miller complement each other well. ConfigForge converts the format into something Miller can process, then Miller does the data work:


# Convert YAML inventory to CSV, then aggregate with Miller
devbench cf inventory.yaml -t csv | mlr --csv stats1 -a count,sum -f quantity -g category

# Convert JSON config to CSV for spreadsheet analysis via Miller
devbench cf config.json -t csv | mlr --csv sort -f name

# Parse a TOML service registry to CSV, filter active services with Miller
devbench cf services.toml -t csv | mlr --csv filter '$status == "active"'

Summary


Miller and ConfigForge solve different problems. Miller is one of the best tools available for streaming CSV/TSV/JSON-lines data processing. ConfigForge covers the config-file formats that Miller ignores: YAML, TOML, HCL, INI, .env, plist, JSONC, XML, and Java .properties.


If you've been using Miller for CSV work and need to also handle YAML or TOML configuration files, ConfigForge is the tool to add to your stack — not a replacement.
