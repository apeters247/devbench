JSONC Converter CLI — tsconfig.json, VS Code Settings, Deno Config


JSONC (JSON with Comments) is JSON extended with // line comments, /* block comments */, and trailing commas. It is used everywhere developers write config files they also need to document:


  • TypeScript tsconfig.json — compiler options with explanatory comments
  • VS Code .vscode/settings.json and keybindings.json
  • Deno deno.jsonc — Deno project configuration
  • Azure DevOps pipeline configs and .devcontainer/devcontainer.json
  • ESLint .eslintrc.jsonc and Prettier configs

The problem: standard JSON parsers reject JSONC files. json.loads() throws a parse error on the first comment. yq returns a cryptic error instead of a useful message. Most online converters silently strip everything after the first //, corrupting values that happen to contain // (URLs, regex patterns).


ConfigForge (devbench cf) supports JSONC as a first-class format with a state-machine comment stripper that correctly handles:


  • String literal preservation — "https://example.com" is never touched; only true comments are stripped
  • Nested block comments — /* outer /* inner */ still outer */
  • Trailing commas — [1, 2, 3,] and {"a": 1,} are parsed correctly
  • Mixed quote styles — double-quoted strings containing // or /* are left intact

Why tsconfig.json Is Hard to Convert


A typical tsconfig.json looks like this:


{
  // Targeting ES2022 for Node 18 compatibility
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "strict": true,  // Enforces noImplicitAny, strictNullChecks, etc.
    /* Output directory — must match Dockerfile COPY path */
    "outDir": "./dist",
    "rootDir": "./src",
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules", "dist"],  // trailing comma OK in JSONC
}

A naive regex approach (re.sub(r'//.*', '', text)) breaks on:


  • "target": "ES2022" — the // in ES2022 would be treated as a comment start if the regex is applied without string-awareness
  • URLs like "baseUrl": "https://cdn.example.com"
  • Regex strings like "exclude": [".*\\.spec\\.ts"] that contain //

ConfigForge uses a character-by-character state machine that tracks quote boundaries, escape sequences, and comment state correctly — the same approach used by TypeScript's own JSON parser.


JSONC to YAML


Converting tsconfig.json to YAML is useful for:


  • Code review tooling — YAML diffs are more readable than JSON diffs for nested config
  • CI/CD matrix configs — some pipelines (GitHub Actions, GitLab) consume YAML more naturally
  • Documentation — YAML with its implicit types is more scannable for humans
  • Cross-tool compatibility — systems that accept YAML but not JSONC

# Input: tsconfig.json (JSONC)
{
  // TypeScript config
  "compilerOptions": {
    "target": "ES2022",
    "strict": true,  // recommended for new projects
  }
}

# Output: tsconfig.yaml
compilerOptions:
  target: ES2022
  strict: true

JSONC to Clean JSON


If you need standard JSON without comments (for tools that only accept RFC 8259 JSON), ConfigForge converts JSONC → JSON by stripping all comments and trailing commas:


devbench cf tsconfig.json --to json
# Outputs: minified or indented standard JSON with zero comments or trailing commas

This is useful for:


  • Sending tsconfig.json content to APIs that parse JSON strictly
  • Generating machine-readable config from human-annotated JSONC source
  • Normalizing JSONC config files before committing to systems that reject comments

VS Code Settings Conversion


VS Code's .vscode/settings.json is always JSONC. It routinely appears in DevOps contexts:


  • Workspace settings committed to repos that need auditing
  • Settings sync scripts that need to compare or diff user vs workspace settings
  • Dev container configs (devcontainer.json) that need to be templated from YAML sources

# Audit all .vscode/settings.json files in a monorepo
devbench cf "./*/.vscode/settings.json" --to yaml --output-dir ./audited-settings/

# Extract a specific setting
devbench cf .vscode/settings.json --get "editor.tabSize"

# Convert to TOML for custom tooling
devbench cf .vscode/settings.json --to toml

Azure DevOps and Bicep


Azure Bicep parameter files (.bicepparam), ARM template comments, and azure-pipelines.yml are used alongside JSONC config files in Azure DevOps workflows. ConfigForge handles the JSONC portions:


# Convert Azure DevOps extension manifest (often JSONC) to YAML
devbench cf vss-extension.json --to yaml

# Inspect parameter values in a JSONC Bicep parameter file
devbench cf parameters.jsonc --get parameters.storageAccountName.value

Format Detection


ConfigForge detects JSONC automatically:


  • Files with a .jsonc extension are always treated as JSONC
  • Files with a .json extension that fail standard JSON parsing are re-tried as JSONC — so tsconfig.json and settings.json are handled without any --from jsonc flag

# .jsonc extension auto-detected
devbench cf deno.jsonc --to yaml

# .json extension auto-detected via content fallback
devbench cf tsconfig.json --to yaml   # works even though it's JSONC
