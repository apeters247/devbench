DevBench / vs yq / Rename Config Key

Rename a YAML / JSON / TOML Key — devbench cf --rename

The problem: A config key needs to be renamed as part of a refactor — server.host → server.hostname, db_password → database.password, or redis.url → cache.url. In any other tool this means writing a custom script, reaching for sed (and getting the quoting wrong), or hand-editing dozens of files. DevBench makes it one flag. 

# Rename a key in a YAML file — output to stdout
devbench cf app.yaml --rename server.host server.hostname

# Edit the file in-place with a backup
devbench cf deploy.yaml --rename spec.replicas spec.replicaCount --in-place --backup

# Move a key to a different parent
devbench cf config.yaml --rename database.password secrets.db_password

# Rename and convert format at the same time
devbench cf app.yaml --rename db.url db.connection_string --to json

# Machine-readable output for scripting
devbench cf app.yaml --rename old_key new_key --raw

Works across all 11 config formats: YAML, JSON, TOML, INI, .env, XML, CSV, HCL, .properties, plist, JSONC. Dot-notation paths for nested keys. In-place editing with optional backup. Fully offline — no API calls.

Why --rename matters in real projects

Config key renames are a constant source of friction in infrastructure work. A Kubernetes Deployment field gets deprecated. An Ansible variable is renamed upstream. A Spring Boot property is reorganized across YAML levels. Each change requires touching the same key in dev, staging, and prod configs — and doing it correctly, without accidentally breaking neighboring keys.

The alternatives are painful:

  • sed — operates on raw text, breaks with complex values, fails on multi-line strings
  • jq — JSON only; requires a function (with_entries(if .key == "old" then .key = "new" else . end))
  • yq — YAML only; no native rename flag; requires yq e 'with_entries(select(.key == "old").key = "new")' file.yaml
  • Python script — fine for one format, not eleven

DevBench parses the config structurally, renames the key, and re-serializes — preserving all other keys, types, and (for YAML) comment metadata wherever possible.

Real-world use cases

1. Kubernetes field migration

Kubernetes deprecated spec.template.spec.hostNetwork in favor of structured networking in newer API versions. Bulk-rename across your cluster manifests: 

# Rename in all deployment manifests
for f in k8s/deployments/*.yaml; do
  devbench cf "$f" --rename spec.replicas spec.replicaCount --in-place --backup .orig
done

2. Spring Boot YAML property migration

Spring Boot 3 renamed spring.datasource.url to spring.datasource.jdbc-url for HikariCP. Update all environment configs at once: 

devbench cf src/main/resources/application.yaml \
  --rename spring.datasource.url spring.datasource.jdbc-url \
  --in-place

3. Terraform variable refactor

Rename an HCL variable across all .tfvars files when your module API changes: 

for f in environments/**/*.tfvars; do
  devbench cf "$f" --rename instance_count replica_count --in-place
done

4. Ansible role variable rename

An Ansible role renamed postgres_db to postgresql_database. Update your inventory: 

devbench cf inventory/group_vars/all.yaml \
  --rename postgres_db postgresql_database \
  --in-place --backup

5. Move a key to a different parent (config restructure)

Restructure a config from a flat layout to grouped namespaces: 

# Before: {db_host: "localhost", db_port: 5432}
devbench cf config.json --rename db_host database.host --rename db_port database.port

6. CI/CD pre-flight: verify rename succeeded

Chain with --assert in a GitHub Actions step: 

- name: Rename deprecated key
  run: devbench cf app.yaml --rename legacy.endpoint api.url --in-place
- name: Verify rename
  run: devbench cf app.yaml --assert api.url=https://api.example.com

Or install extras for HCL and XML:

pip install devbench[all]

All --rename options

devbench cf INPUT --rename OLD_PATH NEW_PATH [options]

Options:
  --in-place, -i        Edit the file in-place (requires a file path)
  --backup [SUFFIX]     Save original to FILE.bak before overwriting
  --to FORMAT           Output format (default: same as input)
  --from FORMAT         Force input format (default: auto-detect)
  --raw                 JSON output: {"success": true, "renamed_from": "...", "renamed_to": "..."}

Dot-notation path examples:
  server.host           Nested key
  containers.0.image    List index
  server\.port          Literal dot in key name (backslash-escape)

Exit codes:
  0   Rename succeeded
  1   Key not found or path error
