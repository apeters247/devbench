ConfigForge vs dasel: 11 Formats + Comments Preserved vs 8 Formats With Comments Dropped


If you found dasel while looking for a multi-format config converter, you're not alone. dasel is a well-regarded Go tool that handles in-place value selection and mutation across JSON, YAML, TOML, XML, CSV, HCL, INI, and KDL. But there's a documented limitation worth knowing about before you commit to it:


"YAML/TOML comments are discarded on write."
— dasel official documentation

This is not a bug dasel is fixing — it's an acknowledged architectural decision. Every time you use dasel to convert or edit a YAML or TOML file, any comments in that file are permanently gone. That's the key difference from ConfigForge (the cf command in the devbench toolkit), which preserves comments through format conversions.


Here's a direct, honest comparison to help you decide.

The Three Formats dasel Doesn't Cover


dasel's 8-format coverage is solid for infrastructure stacks, but modern projects routinely touch three more:


  • .env — Docker Compose, Heroku, GitHub Actions, every Node.js app. The most common config format you never think about until you need to convert it.
  • Apple plist — macOS system preferences, iOS app configs, Xcode build settings. XML and binary plist both supported, cross-platform without Xcode required.
  • JSONC (JSON with Comments) — tsconfig.json, .vscode/settings.json, Azure DevOps pipelines. Standard JSON parsers reject JSONC; ConfigForge strips comments and trailing commas to produce clean JSON, then round-trips back to JSONC.

Also, Java .properties — Spring Boot, Maven, every Java application server. Millions of services still run on these, and dasel has no support.


ConfigForge reads and writes all 11 formats bidirectionally. Converting a Terraform variable file to YAML for an Ansible playbook, or extracting a .env to TOML for a Rust config file — these are one-liners:


# HCL → YAML (e.g. Terraform vars → Ansible vars)
devbench cf convert terraform.tfvars.hcl vars.yaml

# .env → JSON (inject into a JSON-based config system)
devbench cf convert .env config.json

# Java .properties → TOML (modernizing a Spring app)
devbench cf convert application.properties config.toml

# JSONC → JSON (clean tsconfig for tooling that rejects comments)
devbench cf convert tsconfig.json --from jsonc --to json

# Apple plist → YAML (read macOS pref file as human-readable YAML)
devbench cf convert Info.plist settings.yaml

Comment Preservation: dasel Documents This Limitation Itself


This is the most practically important difference for teams that treat their config files as living documentation. dasel's own documentation explicitly states:


"YAML/TOML comments are discarded on write."
— dasel official documentation (permanent architectural decision, not a known bug)

This isn't a bug dasel plans to fix — it's a deliberate trade-off in dasel's architecture. Every YAML and TOML file you process through dasel loses all comments permanently. For config files where the comments explain operational context ("rotate this password every 90 days", "increase this if queue depth exceeds 10k"), that loss is meaningful.


Consider a Terraform variable file with annotations:


# Production DB — rotate every 90 days
db_password = "s3cret"

# Increase this if the queue depth exceeds 10k sustained
worker_count = 4

When you use dasel to convert this to YAML, the comments disappear. When you use ConfigForge:


# Production DB — rotate every 90 days
db_password: s3cret

# Increase this if the queue depth exceeds 10k sustained
worker_count: 4

The comments survive. ConfigForge preserves comments on YAML ↔ TOML ↔ INI ↔ .env round-trips. When target formats don't support comments (JSON, XML, CSV), ConfigForge warns you explicitly instead of silently dropping them — so you always know what metadata was lost and can make an informed decision.


Batch Conversion: Native vs Shell Scripting


Migrating a repository's configuration files from one format to another is a common DevOps task. ConfigForge ships native batch support with glob patterns, recursive directory traversal, and per-file error reporting:


# Convert all YAML files in a tree to JSON
devbench cf convert --batch "**/*.yaml" --to json

# Convert with an output directory (preserves directory structure)
devbench cf convert --batch "configs/**/*.toml" --to yaml --output-dir ./converted/

# Stream mode for 10,000+ files — generator-based, constant memory
devbench cf convert --batch "**/*.json" --to yaml --stream

With dasel, batch work means writing a shell loop:


# dasel approach — manual loop, no error aggregation
find . -name "*.yaml" | while read f; do
  dasel -f "$f" -o json > "${f%.yaml}.json"
done

ConfigForge's batch engine handles glob expansion, output-dir creation, per-file success/failure tracking, and progress reporting. It also provides a streaming mode (--stream) for memory-efficient processing of tens of thousands of files using a constant-memory generator pipeline.


Where dasel Wins: In-Place Editing and Go Binary


dasel has two clear advantages:


In-place selector expressions. dasel has a powerful expression language for reading or mutating specific values within a file without a full format round-trip. For example, dasel put -t string -f config.yaml ".database.host" "localhost" modifies a single key in place. ConfigForge is a full-document converter, not a value editor — it reads the entire document and writes a new one. If you need surgical in-place mutations, dasel is the better tool.


Static Go binary. dasel ships as a single compiled binary with no runtime dependency. If you're deploying to minimal Docker containers or environments without Python, a Go binary is simpler to distribute than a Python package. ConfigForge requires Python 3.10+ and a pip install, which is standard in developer environments but adds friction in stripped-down containers.


Which Tool Should You Use?


Choose dasel if: you need in-place value selection and mutation across JSON/YAML/TOML/XML/CSV, you want a static Go binary with no runtime, and your formats are covered by its five-format support. dasel's selector language is genuinely expressive and worth learning for scripting workflows.


Choose ConfigForge (devbench cf) if: you care about comments surviving format conversions; your project touches .env, plist, JSONC, or Java .properties that dasel doesn't support; you need native batch processing of many files; or you want a unified tool that handles the full 11-format matrix without shell scripting glue. At pip install devbench-cf, it's one command to cover the entire config landscape.


Use both if: you need dasel's selector expressions for in-place edits and ConfigForge's broader format support and batch mode. They complement each other cleanly — dasel for precise mutations, ConfigForge for format migrations and bulk conversions.
