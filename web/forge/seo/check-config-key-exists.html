ConfigForge / Devbench — 11-format config converter

Check If a Config Key Exists

devbench cf config.yaml --path-exists database.password
Exit 0 = key present · Exit 1 = key missing · Works across all 11 config formats

The Problem: Gating on Config Key Presence

Shell scripts often need to check whether optional config keys are set before taking action — TLS configuration, optional database credentials, feature flags. Without a dedicated tool, this requires per-format parsing:

# ❌ Old way — format-specific, fragile grep
if grep -q 'tls:' config.yaml; then echo "TLS configured"; fi
# Breaks on nested keys, TOML input, keys with similar prefixes

# With jq — JSON only
if jq -e '.tls.cert' config.json >/dev/null 2>&1; then echo "TLS"; fi
# With yq — YAML only, no TOML
if yq '.tls.cert' config.yaml | grep -qv null; then echo "TLS"; fi

ConfigForge handles any format with one consistent flag:

# ✅ New way — works on YAML, TOML, JSON, INI, and 8 more formats
if devbench cf config.yaml --path-exists tls.cert; then
    echo "TLS configured — starting secure server"
else
    echo "No TLS cert — starting HTTP server"
fi

Quick Start

pip install devbench

# Check if a key exists (exit 0 = present)
devbench cf config.yaml --path-exists database.host
# EXISTS  database.host
# exit code: 0

# Check for a missing key (exit 1)
devbench cf config.yaml --path-exists database.ssl_cert
# MISSING database.ssl_cert  (on stderr)
# exit code: 1

# JSON output for scripting
devbench cf config.yaml --path-exists tls.cert --raw
# {"path": "tls.cert", "exists": true}

Real-World Use Cases

1. Guard optional TLS configuration

if devbench cf server.yaml --path-exists tls.cert; then
    devbench cf server.yaml --get tls.cert   # get cert path
    openssl verify "$TLS_CERT"
fi

2. Pre-deploy validation in CI/CD

# .github/workflows/deploy.yml
- name: Validate required config keys
  run: |
    devbench cf config/prod.yaml --path-exists database.host
    devbench cf config/prod.yaml --path-exists database.password
    devbench cf config/prod.yaml --path-exists app.secret_key
    echo "All required keys present"

3. Feature flag gating

# Check if a feature flag key exists before reading its value
if devbench cf feature-flags.yaml --path-exists flags.new_dashboard; then
    FLAG=$(devbench cf feature-flags.yaml --get flags.new_dashboard)
    [ "$FLAG" = "true" ] && enable_new_dashboard
fi

4. TOML config (yq can't do this at all)

# pyproject.toml / Cargo.toml style
if devbench cf pyproject.toml --path-exists tool.myapp.cache_dir; then
    CACHE=$(devbench cf pyproject.toml --get tool.myapp.cache_dir)
    mkdir -p "$CACHE"
fi

5. Kubernetes manifest — check optional fields

# Check if resource limits are configured
if devbench cf deployment.yaml --path-exists spec.template.spec.containers[0].resources.limits; then
    echo "Resource limits configured — OK for production"
else
    echo "WARNING: No resource limits set" >&2
    exit 1
fi

6. Structured JSON output for automation

STATUS=$(devbench cf config.yaml --path-exists database.replica --raw)
# {"path": "database.replica", "exists": false}

REPLICA_EXISTS=$(echo "$STATUS" | python3 -c "import sys,json; print(json.load(sys.stdin)['exists'])")
# False
