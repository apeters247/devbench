YAML to TOML Converter — The CLI Tool yq Never Built


If you've tried to convert YAML to TOML using yq, you already know: it doesn't work. yq issue #1364 — "support TOML output" — has been open since 2020 with no resolution. yq can read TOML, but it cannot write it.


devbench cf fills this gap. One command, native TOML output, full type fidelity, 100% offline.


# Convert YAML to TOML — what yq can't do
devbench cf config.yaml -t toml

# From stdin
cat config.yaml | devbench cf -f yaml -t toml

# Batch: convert entire directory
devbench cf "*.yaml" -t toml --out-dir ./toml-output/

# Write to file
devbench cf app.yaml -t toml > app.toml

Why yq Still Has No TOML Output


mikefarah/yq is a Go tool. Its TOML write support has been blocked for years because Go's TOML libraries lack the round-trip precision needed to produce valid TOML from arbitrary YAML. Common failure modes:


  • Inline tables serialized incorrectly
  • YAML merge keys (&anchor / *alias) with no TOML equivalent
  • Null values (~ / null) not representable in TOML
  • Multi-document YAML with no TOML counterpart

devbench cf handles these edge cases explicitly — null values become empty strings or are omitted, anchors are resolved before conversion, and multi-document YAML is split into separate TOML outputs.


Real-World Example: Kubernetes Deployment → TOML


Input: deployment.yaml


apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
  namespace: production
  labels:
    app: my-app
    version: "1.0.0"
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    spec:
      containers:
        - name: app
          image: my-app:1.0.0
          ports:
            - containerPort: 8080
          env:
            - name: ENV
              value: production
            - name: LOG_LEVEL
              value: info

Command:


devbench cf deployment.yaml -t toml

Output: deployment.toml


apiVersion = "apps/v1"
kind = "Deployment"

[metadata]
name = "my-app"
namespace = "production"

[metadata.labels]
app = "my-app"
version = "1.0.0"

[spec]
replicas = 3

[spec.selector.matchLabels]
app = "my-app"

[[spec.template.spec.containers]]
name = "app"
image = "my-app:1.0.0"

[[spec.template.spec.containers.ports]]
containerPort = 8080

[[spec.template.spec.containers.env]]
name = "ENV"
value = "production"

[[spec.template.spec.containers.env]]
name = "LOG_LEVEL"
value = "info"

Example: Python pyproject.yaml → pyproject.toml


Teams that author pyproject config in YAML and need to publish TOML (e.g. for Poetry or Hatch):

  name: my-package
  version: 1.0.0
  description: A Python package
  requires-python: ">=3.10"
  dependencies:
    - requests>=2.28
    - pydantic>=2.0

  requires:
    - hatchling

devbench cf pyproject.yaml -t toml > pyproject.toml

Type Mapping: YAML → TOML


devbench cf preserves native types across the conversion:


YAML type TOML equivalent Notes
string "string" Quoted automatically
integer 42 Native integer
float 3.14 Native float
true / false true / false YAML 1.2 booleans only
null / ~ Key omitted TOML has no null type
sequence (list) [array] or [[table-array]] Objects become table arrays
mapping (dict) [table] Nested sections
&anchor / *alias Resolved inline TOML has no alias concept

Combining with Other Flags


Because devbench cf is a full config manipulation tool, you can transform and convert in one step:


# Extract a nested section and convert to TOML
devbench cf config.yaml --get spec.template -t toml

# Filter a list, then convert matching items to TOML
devbench cf pods.yaml --select status=Running --each name --join , -t toml

# Set a value, then output as TOML
devbench cf config.yaml --set version=2.0.0 -t toml

# Convert while validating required keys exist
devbench cf config.yaml --has version --has name -t toml

Why Not kislyuk/yq?


kislyuk/yq is a Python wrapper around jq — it requires both Python and jq to be installed. It does support TOML output via tomlkit, but:


  • Requires: pip install yq + brew install jq (or apt-get jq)
  • Syntax: cat file.yaml | yq -t '.' > output.toml — jq filter required
  • Not self-contained: jq processes the data; yq is just a format shim
  • No CRUD operations: can't --set, --get, --delete, --merge

devbench cf is self-contained Python, no jq dependency, with full CRUD, filtering, and batch mode:


pip install devbench
devbench cf config.yaml -t toml  # done
