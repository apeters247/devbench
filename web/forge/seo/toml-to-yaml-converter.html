TOML to YAML Converter CLI — The One That Keeps Your Comments


Most config converters silently drop comments when converting TOML to YAML. If your pyproject.toml, Cargo.toml, or config.toml has inline explanations, section headers, or operational notes, they disappear. devbench cf preserves them.


# Install
pip install devbench

# Basic TOML → YAML
devbench cf config.toml -t yaml

# Write to a file
devbench cf config.toml -t yaml > config.yaml

# Pipe from stdin
cat pyproject.toml | devbench cf -f toml -t yaml

# In-place conversion
devbench cf config.toml -t yaml --in-place

Comment Preservation: The Difference That Matters


Here's a typical TOML configuration with operational comments:


# Database connection — rotate credentials every 90 days
[database]
host = "db.internal"
port = 5432  # non-standard port for firewall rule
# Increase pool size if p99 latency > 200ms
pool_size = 10

[cache]
# Redis — requires AUTH in production
url = "redis://cache:6379"
ttl = 3600

With most tools (yq, dasel, online converters):


database:
  host: db.internal
  port: 5432
  pool_size: 10
cache:
  url: redis://cache:6379
  ttl: 3600

The operational context is gone. With devbench cf:


# Database connection — rotate credentials every 90 days
database:
  host: db.internal
  port: 5432  # non-standard port for firewall rule
  # Increase pool size if p99 latency > 200ms
  pool_size: 10

# Redis — requires AUTH in production
cache:
  url: redis://cache:6379
  ttl: 3600

Comments survive the round-trip. Your documentation stays with your data.


Common Migration Patterns


Rust / Cargo.toml to YAML (Ansible/Kubernetes)

# Extract dependencies section from Cargo.toml as YAML
devbench cf Cargo.toml --get dependencies -t yaml

# Convert full Cargo.toml for documentation generation
devbench cf Cargo.toml -t yaml > cargo-manifest.yaml

Python / pyproject.toml to YAML

# Extract project metadata
devbench cf pyproject.toml --get project -t yaml

# Extract tool.pytest config for CI YAML step
devbench cf pyproject.toml --get tool.pytest.ini_options -t yaml

Hugo / config.toml to YAML

# Hugo supports both formats — migrate from TOML to YAML
devbench cf config.toml -t yaml > config.yaml

# Verify structure before switching
devbench cf config.toml -t yaml | devbench cf -f yaml --validate

Batch Conversion


# Convert all TOML files in a directory
devbench cf "*.toml" --batch -t yaml

# Recursive — all TOML files in a project tree
devbench cf "**/*.toml" --batch -t yaml

# With output directory
devbench cf "configs/**/*.toml" --batch -t yaml --output-dir ./yaml-configs/

# Preview what would be converted (dry run via --validate)
devbench cf "**/*.toml" --batch --validate

TOML Types in YAML


TOML has explicit typing that YAML handles through type inference. devbench cf maps them correctly:


TOML TypeTOML ValueYAML Output
String"hello"hello
Integer4242
Float3.143.14
Booleantruetrue
Datetime1979-05-27T07:32:00Z'1979-05-27T07:32:00+00:00'
Array[1, 2, 3]- 1\n- 2\n- 3
Table[section]YAML mapping block

Reverse: YAML to TOML


The reverse direction works equally well:


# YAML → TOML
devbench cf config.yaml -t toml

# YAML → TOML with sorted keys (for readable Cargo.toml output)
devbench cf deps.yaml -t toml --sort-keys

devbench cf is one of the few Python tools that supports TOML write. The tomllib module in Python's stdlib is read-only; devbench cf uses tomli_w to write TOML correctly.
