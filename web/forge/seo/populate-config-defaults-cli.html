Populate Config Defaults Without Overwriting Existing Values


A common CI/CD pattern: you have a team-wide defaults.yaml and per-environment config files. You want to ensure every config file has all the required keys — but you never want to clobber values that someone has already set. Standard --merge always lets the overlay win. devbench cf --merge-new-only inverts this: the base always wins, and only keys absent from the base are added from the defaults file.


devbench cf config.yaml --merge defaults.yaml --merge-new-only

This directly implements yq issue #2201 ("merge only keys that don't exist in base").


Basic Example


# config.yaml — existing environment-specific config
database:
  host: db.prod.example.com
  port: 5432


# defaults.yaml — team-wide defaults for every key
database:
  host: localhost
  port: 5432
  pool_size: 10
  timeout: 30
  ssl: true


devbench cf config.yaml --merge defaults.yaml --merge-new-only

database:
  host: db.prod.example.com   # preserved — already set
  port: 5432                   # preserved — already set
  pool_size: 10                # added from defaults
  timeout: 30                  # added from defaults
  ssl: true                    # added from defaults

The existing host and port are never touched. Only pool_size, timeout, and ssl — which were absent from the base — are injected.


CI/CD: Ensure Required Keys Are Present


#!/bin/bash
# Ensure every service config has all required observability keys
DEFAULTS=config/observability-defaults.yaml

for f in config/services/*.yaml; do
  devbench cf "$f" \
    --merge "$DEFAULTS" \
    --merge-new-only \
    --in-place
done

This adds any missing observability keys to every service config without touching service-specific values already defined.


Populate Defaults at a Specific Nested Path


Combine --merge-new-only with --merge-at to inject defaults only into a specific section of the config:


# base.yaml
db:
  host: localhost
  port: 5432


# db-defaults.yaml — just the key names and fallback values
host: db.prod
port: 9999
timeout: 30
max_connections: 20


devbench cf base.yaml \
  --merge db-defaults.yaml \
  --merge-at db \
  --merge-new-only

db:
  host: localhost        # preserved
  port: 5432             # preserved
  timeout: 30            # added
  max_connections: 20    # added

The defaults file contains only the default values — no nesting required. --merge-at db targets the right section. --merge-new-only ensures existing production values are never overwritten.


Cross-Format: JSON Config + YAML Defaults


# app.json gets any missing keys from YAML defaults
devbench cf app.json --merge defaults.yaml --merge-new-only

# Output as JSON (format preserved automatically)
# Or force a specific format:
devbench cf app.json --merge defaults.yaml --merge-new-only --to yaml

Verify What Would Be Added (Dry Run)


# See what keys are missing before applying
devbench cf config.yaml --diff defaults.yaml

# Apply when ready
devbench cf config.yaml --merge defaults.yaml --merge-new-only --in-place

Use Cases


  • Defaults files in monorepos: ensure every service config has required telemetry, logging, or feature flag keys without clobbering service-specific values.
  • Config migrations: add new required keys introduced in a new version without touching configs that already have custom values.
  • Kubernetes overlays: inject missing resources.limits or probes into deployments that don't define them yet, without disturbing deployments that already do.
  • Onboarding new environments: start from a full defaults file and let per-environment overrides take precedence automatically.
