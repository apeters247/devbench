Inject YAML Under a Nested Path from the Command Line


A recurring complaint on r/devops and in yq GitHub issues: "you cannot add a chunk of YAML file under .spec.containers[]". Most config tools — including yq — only merge overlays at the document root. If you want to inject tolerations into spec.template.spec of a Kubernetes deployment, you're forced to write a multi-step expression or repeat the entire path in your patch file.


devbench cf --merge-at PATH targets any nested path for the overlay merge, leaving the rest of the document untouched.


The Problem: yq Merges at Root Only


With yq, merging a tolerations snippet into spec.template.spec requires knowing and repeating the full path in the patch file:


# patch.yaml must mirror the full nesting — yq merges at root
spec:
  template:
    spec:
      tolerations:
        - key: gpu
          effect: NoSchedule

This means your patch file is 95% structural boilerplate and 5% actual change. Any path typo silently merges at the wrong level.


The Fix: devbench cf --merge-at


Write only the content that changes. Point --merge-at at the target path:


# tolerations.yaml — just the new content, no nesting required
tolerations:
  - key: gpu
    effect: NoSchedule
  - key: spot-instance
    effect: NoSchedule
    operator: Exists


devbench cf deploy.yaml \
  --merge tolerations.yaml \
  --merge-at spec.template.spec

# Output — tolerations injected, nothing else touched
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  template:
    spec:
      nodeSelector:
        kubernetes.io/arch: amd64
      tolerations:         # ← injected
        - key: gpu
          effect: NoSchedule
        - key: spot-instance
          effect: NoSchedule
          operator: Exists
      containers:
        - name: app
          image: myapp:v2

More Examples


Add DB connection defaults without clobbering existing config


# defaults.yaml
timeout: 30
pool_size: 10
ssl: true


devbench cf config.yaml \
  --merge defaults.yaml \
  --merge-at db \
  --merge-new-only

Combines --merge-at with --merge-new-only: existing db.timeout or db.pool_size values are preserved; only keys absent from the base are added. See populate-config-defaults-cli for more on --merge-new-only.


Inject logging config into a nested app section


# logging-patch.yaml
level: warn
format: json
output: /var/log/app.log


devbench cf app.yaml --merge logging-patch.yaml --merge-at app.logging

Create the target path if it doesn't exist


--merge-at creates intermediate paths that don't exist yet — no pre-existing structure required:


# base.yaml
name: myservice
version: "1.0"


devbench cf base.yaml --merge observability.yaml --merge-at telemetry.otel

name: myservice
version: "1.0"
telemetry:
  otel:            # ← created from scratch
    endpoint: http://collector:4317
    sampling: 0.1

Compose with --in-place for CI Patching


# Patch tolerations into every deployment in a loop
for f in k8s/deployments/*.yaml; do
  devbench cf "$f" \
    --merge k8s/patches/gpu-tolerations.yaml \
    --merge-at spec.template.spec \
    --in-place
done

# Pipe directly to kubectl
devbench cf deploy.yaml \
  --merge prod-overrides.yaml \
  --merge-at spec.template.spec \
  | kubectl apply -f -

Install


pip install devbench
devbench cf --help

Works on Python 3.10–3.13. Fully offline — no telemetry, no external API calls. Supports 11 formats: JSON, YAML, TOML, XML, CSV, INI, .env, HCL, Java .properties, Apple plist, JSONC.
