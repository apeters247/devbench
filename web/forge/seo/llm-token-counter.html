CLI Token Counter for LLMs + Text Chunker for RAG


If you're building LLM applications, you need two things regularly: how many tokens is this text? and how do I split this document into chunks that fit my context window? Most developers write throwaway Python scripts for both. Devbench ships both as first-class CLI tools — devbench token and devbench chunk.


Token Counter (devbench token)


Count tokens for any text using tiktoken — the same tokenizer used by GPT-4, GPT-3.5-turbo, and Claude models.


# Count tokens in a string
echo "Hello, world!" | devbench token

# Count tokens in a file
devbench token < prompt.txt

# Count tokens in a long system prompt
cat system-prompt.md | devbench token

Example output:

{
  "tool": "token",
  "output": "4",
  "metadata": {
    "model": "cl100k_base",
    "token_count": 4,
    "char_count": 13
  },
  "success": true,
  "error": null
}

The cl100k_base encoding is used by GPT-4, GPT-3.5, and Claude models via the tiktoken library. When tiktoken is not installed, Devbench falls back to a character-based estimate (approximately 4 chars per token) and notes the estimate in the output.


Why a dedicated token counter?


The common Python one-liner for counting tokens is:

python3 -c "import tiktoken; enc = tiktoken.get_encoding('cl100k_base'); print(len(enc.encode(open('file.txt').read())))"

That's 90 characters you have to remember, re-type, or copy-paste every time. devbench token < file.txt is 24 characters and returns structured JSON you can pipe into jq:


# Is this prompt over 4096 tokens?
devbench token < long-prompt.txt | jq '.metadata.token_count > 4096'

# Get just the count as a number
devbench token < docs.txt | jq -r '.metadata.token_count'

Text Chunker for RAG (devbench chunk)


Split long documents into overlapping chunks for retrieval-augmented generation (RAG). The chunker is sentence-aware — it doesn't split mid-sentence — with configurable chunk size (in tokens) and overlap.


# Default: 500-token chunks with 100-token overlap
cat docs/architecture.md | devbench chunk

# Smaller chunks for embedding with dense retrieval
cat paper.txt | devbench chunk --chunk-size 256 --chunk-overlap 50

# No overlap (non-overlapping windows)
cat transcript.txt | devbench chunk --chunk-size 1000 --chunk-overlap 0

Example output for a multi-paragraph document:

{
  "tool": "chunk",
  "output": "3 chunks",
  "metadata": {
    "chunk_count": 3,
    "chunk_size": 500,
    "chunk_overlap": 100,
    "chunks": [
      { "index": 0, "text": "Chapter 1: Introduction\n\nThis document covers...", "token_estimate": 487 },
      { "index": 1, "text": "...continued from the introduction. The core idea...", "token_estimate": 512 },
      { "index": 2, "text": "...In conclusion, the system provides...", "token_estimate": 201 }
    ]
  },
  "success": true
}

Sentence-aware splitting


Most naive chunkers split on fixed character counts, cutting mid-sentence. Devbench's chunker splits on sentence boundaries so each chunk is coherent and readable — important for embedding quality and LLM context coherence.

Installation


# Install with tiktoken for accurate token counts
pip install devbench-cf[tiktoken]

# Or install the base (uses character-based estimate)
pip install devbench-cf

Both devbench token and devbench chunk work on macOS, Linux, and Windows. No API key, no network calls, no data leaves your machine.
