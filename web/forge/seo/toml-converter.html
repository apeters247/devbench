YAML to TOML Converter — The CLI yq Users Keep Asking For


If you've ever searched for a YAML-to-TOML converter and landed on a yq issue thread, you already know the problem. yq is a powerful YAML processor, but it cannot write TOML output. From the official yq documentation:


"yq does not yet support outputting in TOML format and therefore cannot roundtrip TOML."
— yq (kislyuk) official documentation

This limitation has been open for years. ConfigForge fills that gap: full bidirectional TOML conversion across all 11 supported formats, from the command line, offline, cross-platform.


Quick Start


# Install (Python 3.10–3.13, macOS/Linux/Windows)
pip install devbench-cf

# YAML → TOML
cf convert config.yaml config.toml

# JSON → TOML
cf convert package.json config.toml

# HCL → TOML (Terraform vars to Rust config)
cf convert terraform.tfvars.hcl config.toml

# INI → TOML (legacy config modernization)
cf convert app.ini config.toml

# Stdin → TOML (pipe from another tool)
cat config.yaml | cf --from yaml --to toml

# TOML → YAML (full round-trip)
cf convert Cargo.toml cargo.yaml

Supported TOML Conversions


ConfigForge reads any of its 11 formats and writes TOML, and reads TOML and writes to any other format. Bidirectional, bidirectional.


Input Format→ TOML OutputTOML Input →Notes
YAML (.yaml, .yml)✅✅Comment preservation on YAML ↔ TOML round-trip
JSON (.json)✅✅JSON null → TOML omitted (documented)
JSONC (.jsonc, tsconfig.json)✅✅Strips JSONC comments before converting
INI (.ini, .cfg)✅✅INI sections → TOML tables
.env (dotenv)✅✅All values as TOML strings unless typed
HCL (.hcl, .tfvars)✅✅Terraform variable files → TOML
Java .properties✅✅Dotted keys flattened to TOML tables
XML (.xml)✅✅Attributes as TOML keys
CSV (.csv)✅✅Array of tables (one row per TOML table)
Apple plist (.plist)✅✅XML and binary plist both read correctly

Common Use Cases


Rust project: migrating from YAML CI config to TOML


Rust projects use TOML natively (Cargo.toml), but CI tools often default to YAML. ConfigForge handles the migration:


# Convert GitHub Actions config sections to TOML for local tooling
cf convert .github/workflows/ci.yaml ci-config.toml

# Extract values from Cargo.toml into YAML for Ansible
cf convert Cargo.toml --to yaml | grep version

pyproject.toml: migrating from setup.cfg or requirements-style configs


Modern Python packaging uses pyproject.toml. If you have an old setup.cfg or INI-style config, convert it directly:


# setup.cfg (INI format) → pyproject.toml fragment
cf convert setup.cfg --to toml

# application.ini → TOML for newer Python tooling
cf convert application.ini config.toml

Hugo static site: managing config in YAML, deploying in TOML


Hugo supports both YAML and TOML config files. Some teams prefer editing in YAML (comments!) and deploying in TOML (Hugo's default). ConfigForge preserves comments on the YAML source and writes clean TOML:


# YAML source (editable, with comments) → TOML deploy artifact
cf convert hugo-config.yaml hugo.toml

# Round-trip check: TOML back to YAML
cf convert hugo.toml --to yaml

Batch convert an entire repository


Migrating a project from YAML configs to TOML across dozens of files:


# Convert all YAML files recursively, output to ./toml-configs/
cf convert --batch "**/*.yaml" --to toml --output-dir ./toml-configs/

# Stream mode for large repositories (constant memory, generator-based)
cf convert --batch "configs/**/*.yaml" --to toml --stream

Why Not yq?


yq is an excellent YAML processor. But the TOML limitation is real and multi-year. From the yq GitHub issues, the tomlq shim only converts TOML→JSON; it cannot generate TOML output. For any workflow that requires writing TOML — not just reading it — yq is the wrong tool. ConfigForge was built to fill this gap.


A side-by-side:


# yq: YAML → TOML (does not work)
yq -o toml config.yaml    # ERROR: yq cannot output TOML

# ConfigForge: YAML → TOML (works)
cf convert config.yaml config.toml    # ✅ done

TOML Write: Comment Preservation Detail


When you convert YAML to TOML, ConfigForge attempts to preserve block comments at the start of values where the TOML format supports them:


# config.yaml — YAML source with comments
# Maximum concurrent workers — tuned for m5.2xlarge
max_workers: 8

# Database connection (read replica for analytics queries)
database_url: "postgres://replica.internal:5432/analytics"


# config.toml — output from cf convert config.yaml config.toml

# Maximum concurrent workers — tuned for m5.2xlarge
max_workers = 8

# Database connection (read replica for analytics queries)
database_url = "postgres://replica.internal:5432/analytics"

Your operational comments survive the round-trip. dasel, by documented design, drops them.
