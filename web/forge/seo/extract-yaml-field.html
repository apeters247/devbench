The Problem with yq and jq for Field Extraction

Every DevOps engineer has been there: you want to pull a single value from a Kubernetes YAML manifest and you end up googling "yq get field syntax" for the third time this week. 

# yq v4 — remember which expression syntax?
yq '.spec.replicas' deploy.yaml
yq eval '.spec.template.spec.containers[0].image' deploy.yaml

# jq — only works on JSON, quotes required
jq -r '.database.host' config.json

# Extract two fields with jq? Need to construct an object:
jq '{host: .database.host, port: .database.port}' config.json

For a single value these tools are fine. For multiple fields, or for non-JSON formats, the filter expression syntax becomes friction. You're building a new object literal instead of just naming the fields you want.

ConfigForge --pick: Name the Fields You Want

--pick PATH [PATH ...] takes one or more dot-notation paths and outputs just those fields. No filter expressions. No format-specific syntax. Works on all 11 supported formats. 

pip install devbench

# Extract a single value from YAML
devbench cf deploy.yaml --pick spec.replicas
# → 3

# Extract multiple fields — output is a new YAML with just those keys
devbench cf deploy.yaml --pick spec.replicas metadata.name metadata.namespace

# Cross-format: extract from YAML, output as JSON
devbench cf deploy.yaml --pick spec.replicas metadata.name --to json

# Extract from any supported format
devbench cf config.toml --pick server.host server.port --to json
devbench cf app.ini --pick database.url cache.ttl
devbench cf .env --pick DATABASE_URL REDIS_URL --to yaml

Single Field: Bare Value Output

When you pick exactly one path, ConfigForge outputs the raw value — no wrapping object:

devbench cf deploy.yaml --pick spec.replicas
# → 3

# Perfect for shell scripts:
REPLICAS=$(devbench cf deploy.yaml --pick spec.replicas)
IMAGE=$(devbench cf deploy.yaml --pick spec.template.spec.containers[0].image)
echo "Scaling $IMAGE to $REPLICAS replicas"

Multiple Fields: Projected Config

When you pick two or more paths, ConfigForge builds a new config containing only those fields. The paths become keys in the output — flat, readable, unambiguous: 

# Input: deploy.yaml (200 lines)
devbench cf deploy.yaml --pick spec.replicas metadata.name spec.template.spec.containers[0].image --to json
# →
# {
#   "spec.replicas": 3,
#   "metadata.name": "web-deployment",
#   "spec.template.spec.containers[0].image": "nginx:1.25"
# }

# Output as YAML:
devbench cf deploy.yaml --pick spec.replicas metadata.name
# →
# spec.replicas: 3
# metadata.name: web-deployment

Real-World Use Cases

1. Kubernetes Manifest Inspection

# What image is deployed?
devbench cf k8s/production/deploy.yaml --pick spec.template.spec.containers[0].image

# Compare replica counts across environments
for env in staging production; do
  echo -n "$env: "
  devbench cf "k8s/$env/deploy.yaml" --pick spec.replicas
done
# → staging: 2
# → production: 10

# Extract resource limits to audit over-provisioning
devbench cf deploy.yaml \
  --pick spec.template.spec.containers[0].resources.requests.cpu \
         spec.template.spec.containers[0].resources.limits.memory \
  --to json

2. Extract Database Config for a Script

# Pull just the database section fields
DB_HOST=$(devbench cf config.yaml --pick database.host)
DB_PORT=$(devbench cf config.yaml --pick database.port)
DB_NAME=$(devbench cf config.yaml --pick database.name)

pg_dump -h "$DB_HOST" -p "$DB_PORT" "$DB_NAME" > backup.sql

3. Cross-Format Field Extraction

# Terraform .tfvars → extract specific variables for a script
devbench cf terraform.tfvars --pick aws_region instance_type --to json

# Docker Compose → extract service image names
devbench cf docker-compose.yaml --pick services.web.image services.db.image

# Ansible inventory → extract hosts
devbench cf inventory.ini --pick webservers.hosts staging.hosts --to json

4. Config Audit: Check Required Keys Exist

#!/bin/bash
# Verify required fields are present in a config
REQUIRED_FIELDS="database.host database.port app.secret_key"
for field in $REQUIRED_FIELDS; do
  if ! devbench cf config.yaml --pick "$field" &>/dev/null; then
    echo "MISSING: $field"
    exit 1
  fi
done
echo "All required fields present"

5. Generate a Summary Report

# Extract key deployment metadata as JSON for a dashboard
devbench cf deploy.yaml \
  --pick metadata.name metadata.namespace spec.replicas \
         spec.template.spec.containers[0].image \
  --to json | curl -X POST https://dashboard.internal/api/deployments -d @-

When to use yq: You're already deep in a YAML-only pipeline, you need complex filter expressions (conditionals, map/select), or you're processing very large files where yq's Go performance matters.

When to use ConfigForge --pick: You work across multiple config formats, you want readable flag-based syntax, you need the extracted values in a different format (e.g., pick from YAML, output as TOML), or you're building shell scripts that need bare values without quoting gymnastics.
