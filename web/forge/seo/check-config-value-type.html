Check Config Value Type from the CLI

devbench cf config.yaml --type server.port outputs the JSON Schema type of any config value: string, integer, number, boolean, array, object, or null. Works across all 11 supported config formats. No jq, no Python script, no awk pipeline.

Quick Examples

# Simple type check
$ devbench cf app.yaml --type server.port
integer

$ devbench cf app.yaml --type server.host
string

$ devbench cf app.yaml --type features
array

$ devbench cf app.yaml --type database
object

# JSON output with length for containers
$ devbench cf app.yaml --type database --raw
{"path": "database", "type": "object", "length": 3}

$ devbench cf app.yaml --type tags --raw
{"path": "tags", "type": "array", "length": 5}

# Check root value type
$ devbench cf app.yaml --type .
object

# Use in a shell script
TYPE=$(devbench cf config.yaml --type replicas)
if [ "$TYPE" = "integer" ]; then
    echo "Replicas is an integer: OK"
else
    echo "ERROR: replicas must be an integer, got $TYPE" >&2
    exit 1
fi

Why You Need This

Config type bugs are notoriously silent. A YAML file with port: "8080" (string) instead of port: 8080 (integer) passes YAML parsing but breaks downstream applications that call int(config.port). Docker Compose accepts both forms but Kubernetes rejects the string version. Helm charts fail silently when values are the wrong type.

The classic workaround is a jq expression or a tiny Python script. Both require knowing the target format. devbench cf --type works across all 11 config formats with one consistent interface.

Note: DevBench correctly distinguishes boolean from integer, even though Python's bool is a subclass of int. A YAML value of true reports boolean; a value of 1 reports integer. This matches JSON Schema semantics exactly.

Machine-Readable Output (--raw)

# Array type with length
$ devbench cf k8s/deployment.yaml --type spec.template.spec.containers --raw
{"path": "spec.template.spec.containers", "type": "array", "length": 2}

# Scalar — no length field
$ devbench cf k8s/deployment.yaml --type spec.replicas --raw
{"path": "spec.replicas", "type": "integer"}

# Error case — machine-readable
$ devbench cf config.yaml --type missing.key --raw
{"path": "missing.key", "error": "'missing'"}
# exit code: 1

CI/CD Type Validation Pattern

#!/bin/bash
# Validate config types before deployment
set -e

fail() { echo "CONFIG ERROR: $1" >&2; exit 1; }

check_type() {
    local file=$1 path=$2 expected=$3
    actual=$(devbench cf "$file" --type "$path") || fail "$path not found"
    [ "$actual" = "$expected" ] || fail "$path must be $expected, got $actual"
}

check_type config/prod.yaml database.port integer
check_type config/prod.yaml database.ssl_enabled boolean
check_type config/prod.yaml allowed_hosts array
check_type config/prod.yaml feature_flags object

echo "All config types valid ✓"

Works with Any Supported Format

# TOML
$ devbench cf app.toml --type server.port
integer

# .properties file
$ devbench cf db.properties --type database.pool_size
string  # .properties values are always strings

# HCL (Terraform)
$ devbench cf main.tf --type variable.instance_count.default
integer

# INI
$ devbench cf settings.ini --type database.port
string  # INI values parsed as strings by default

# Combine with --to for cross-format output
$ devbench cf config.toml --type features --raw --to json
{"path": "features", "type": "array", "length": 4}
