CloudFormation YAML to JSON Converter — Offline, Comment-Safe, Batch-Ready


AWS CloudFormation supports both YAML and JSON template formats, and real-world infrastructure stacks often contain both — legacy JSON templates mixed with newer YAML, or tool outputs in one format that your pipeline expects in the other. Converting between them manually is tedious and error-prone. Online converters require you to paste your entire template (including secrets, resource ARNs, and account IDs) into a browser.


ConfigForge (the cf command in devbench) converts CloudFormation templates between YAML and JSON locally, with no data leaving your machine. It handles complex nested structures, preserves document structure, and supports batch conversion of entire stacks.


Quick Start


# Install
pip install devbench-cf

# YAML → JSON
devbench cf convert template.yaml template.json

# JSON → YAML (YAML is more readable for complex templates)
devbench cf convert template.json template.yaml

# Auto-detect format from file extension
devbench cf convert stack.yaml --to json

Why Convert CloudFormation Templates?


There are several common reasons to convert between the two formats:


  • Migrating legacy JSON stacks to YAML — YAML is more readable for complex templates with hundreds of resources. Inline comments, multi-line strings, and shorter syntax make YAML the preferred format for maintenance.
  • Tooling compatibility — some tools (older CDK versions, certain CI/CD integrations, deployment scripts) expect a specific format. Converting to match is faster than patching the tooling.
  • Diff readability — YAML diffs are significantly more readable than JSON diffs in pull requests, especially for deeply nested CloudFormation resource properties.
  • Template linting — some CloudFormation linters and validators work better with one format over the other.
  • Sharing and documentation — YAML is easier for humans to read and annotate with comments.

Handling CloudFormation Intrinsic Functions


CloudFormation's YAML syntax uses shorthand intrinsic functions (!Ref, !Sub, !Join, !Select, !If, etc.) that are YAML tags. These serialize correctly to their long-form JSON equivalents ({"Ref": ...}, {"Fn::Sub": ...}) and back.


# YAML — shorthand syntax
Resources:
  MyBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: !Sub "${AWS::StackName}-data-${AWS::Region}"
      Tags:
        - Key: Environment
          Value: !Ref EnvironmentParam

After conversion to JSON:


{
  "Resources": {
    "MyBucket": {
      "Type": "AWS::S3::Bucket",
      "Properties": {
        "BucketName": {
          "Fn::Sub": "${AWS::StackName}-data-${AWS::Region}"
        },
        "Tags": [
          {
            "Key": "Environment",
            "Value": { "Ref": "EnvironmentParam" }
          }
        ]
      }
    }
  }
}

The conversion preserves the full structure and all intrinsic function semantics. CloudFormation's validator accepts both representations interchangeably.


Converting Comments — The Privacy Argument


CloudFormation templates for production infrastructure often contain sensitive annotations: account IDs, ARNs, operational runbooks, security group rationale, and cost center codes. Pasting these into a browser-based converter is a security risk. Even "free" online tools may log, cache, or train on your input.


ConfigForge runs entirely locally. Nothing leaves your machine. No API calls, no telemetry, no logging. It's the right tool when your templates contain anything you wouldn't want in a stranger's access log.


When converting YAML → JSON, ConfigForge warns you that JSON doesn't support comments (it can't), so you know your inline annotations are being preserved in the YAML source and are not present in the JSON output. When converting back JSON → YAML, the structure is clean and ready for you to annotate.


Batch Converting an Entire Stack


Large projects have many templates — nested stacks, per-service templates, shared infrastructure stacks. ConfigForge's batch mode converts a whole tree at once:


# Convert all YAML templates in a stacks/ directory to JSON
devbench cf convert --batch "stacks/**/*.yaml" --to json --output-dir ./json-stacks/

# Convert all JSON templates to YAML (for a readability migration)
devbench cf convert --batch "templates/**/*.json" --to yaml --output-dir ./yaml-templates/

# Preview what would be converted (dry-run via glob expansion)
devbench cf convert --batch "**/*.yaml" --to json --dry-run

The output directory mirrors the source directory structure, so stacks/vpc/main.yaml becomes json-stacks/vpc/main.json. Per-file errors are reported individually — a malformed template doesn't abort the entire batch.


Combining With cfn-lint


A common pipeline pattern is to convert to YAML for human review, then convert back to JSON for deployment (if your pipeline requires JSON). With ConfigForge, this is a two-step round-trip that preserves all data:


# Standard workflow: edit in YAML, deploy in JSON
# 1. Convert to YAML for review and editing
devbench cf convert template.json template.yaml

# 2. Edit template.yaml, add comments, review in PR

# 3. Lint the YAML (cfn-lint supports YAML natively)
cfn-lint template.yaml

# 4. Convert back to JSON for deployment
devbench cf convert template.yaml template-deploy.json

# 5. Deploy
aws cloudformation deploy --template-file template-deploy.json ...

CloudFormation Template Size and Performance


CloudFormation templates can be large — 500+ resource templates are not unusual in mature infrastructure projects. ConfigForge handles these efficiently:


  • Single large files: ConfigForge reads the entire file into memory for format conversion, which is fast for files up to several MB. Python's YAML and JSON parsers are well-optimized for the sizes CloudFormation produces.
  • Batch / streaming: For repositories with hundreds of templates, use --batch --stream to convert with constant memory usage regardless of the total file count.
  • No size limit: ConfigForge has no artificial size cap. CloudFormation's own 51,200-byte template size limit means you'll rarely hit performance issues.
