Assert Config Values in CI/CD — devbench cf --assert

The problem: You want to verify a config key equals a specific value in a CI/CD pipeline — that spec.replicas is 3, that server.debug is false before deploying to production, that a database port hasn't drifted. In every other tool this takes a jq expression, a shell comparison, or a custom script. DevBench makes it one flag. 

# Exit 0 if replicas == 3, exit 1 otherwise
devbench cf deploy.yaml --assert spec.replicas=3

# Assert multiple keys — exit 1 if any fail
devbench cf config.yaml --assert db.host=prod-db --assert db.ssl=true

# Machine-readable output for CI logs
devbench cf app.toml --assert server.debug=false --raw

Works across all 11 config formats: YAML, JSON, TOML, INI, .env, XML, CSV, HCL, .properties, plist, JSONC. No dependencies beyond pip install devbench. Runs fully offline.

Why --assert matters in DevOps pipelines

Config drift is a real production risk. A Kubernetes deployment accidentally scaled to 1 replica instead of 3. A debug flag left true in a staging config promoted to production. A database password rotated in secrets manager but not in the config file. These failures are cheap to catch at deploy time — if you have a way to assert the values are correct.

With standard tools, asserting a config value requires awkward pipelines: 

# The jq way — JSON only, verbose
REPLICAS=$(jq '.spec.replicas' deploy.json)
if [ "$REPLICAS" != "3" ]; then echo "FAIL"; exit 1; fi

# The yq way — YAML only, version-sensitive syntax
yq '.spec.replicas' deploy.yaml | grep -q '^3$' || exit 1

# ConfigForge — any format, one command
devbench cf deploy.yaml --assert spec.replicas=3

Type-aware comparison

ConfigForge coerces the expected value using the same JSON-aware logic as --set, so --assert replicas=3 compares against the integer 3, not the string "3". Booleans, nulls, and floats all work: 

devbench cf config.yaml --assert server.debug=false   # boolean false
devbench cf config.yaml --assert server.workers=4     # integer 4
devbench cf config.yaml --assert database.timeout=30  # integer 30
devbench cf config.yaml --assert optional_key=null    # null/None

Real-world use cases

1. Kubernetes deployment guard

Verify minimum replicas are set before applying a manifest:

# In .github/workflows/deploy.yml
- name: Assert production replicas
  run: |
    devbench cf k8s/production/deployment.yaml --assert spec.replicas=3
    devbench cf k8s/production/deployment.yaml --assert spec.template.spec.containers[0].resources.requests.memory=512Mi

2. Pre-commit hook: no debug in production configs

#!/bin/sh
# .git/hooks/pre-commit
for f in config/prod*.yaml; do
  devbench cf "$f" --assert server.debug=false || {
    echo "ERROR: debug=true found in $f — aborting commit"
    exit 1
  }
done

3. Docker Compose port assertion

devbench cf docker-compose.yml --assert services.web.ports[0]=80:80
devbench cf docker-compose.yml --assert services.db.ports[0]=5432:5432

4. Multi-key CI check with JSON output

# In GitHub Actions, capture structured output
RESULT=$(devbench cf config/production.yaml \
  --assert database.host=prod-db.example.com \
  --assert database.ssl=true \
  --assert cache.ttl=3600 \
  --raw)
echo "$RESULT" | jq .
# {"assertions": [...], "all_passed": true}

5. Terraform variable assertion

# Verify instance type before apply
devbench cf terraform.tfvars --assert instance_type=t3.medium || {
  echo "Wrong instance type in production vars!"
  exit 1
}

6. Ansible inventory check

# Assert environment tag is set correctly
devbench cf inventory/production.yaml --assert all.vars.env=production
devbench cf inventory/production.yaml --assert all.vars.region=us-east-1

Exit codes and shell integration

Exit codes follow Unix assertion conventions — ideal for && chains and CI/CD gates:

Exit codeMeaning
0All assertions passed
1One or more assertions failed, key missing, or parse error
 
# Short-circuit on first failure
devbench cf config.yaml --assert version=1.0.0 \
  && devbench cf config.yaml --assert debug=false \
  && deploy.sh

# Or check all at once, fail if any fail
devbench cf config.yaml \
  --assert version=1.0.0 \
  --assert debug=false \
  --assert replicas=3
echo "Exit: $?"

JSON output for CI logs

Add --raw (or -r) for machine-readable JSON — useful for structured CI logs or downstream scripting:

$ devbench cf deploy.yaml \
    --assert spec.replicas=3 \
    --assert spec.image=nginx:1.21 \
    --raw
{
  "assertions": [
    {
      "path": "spec.replicas",
      "expected": 3,
      "actual": 3,
      "passed": true,
      "missing": false
    },
    {
      "path": "spec.image",
      "expected": "nginx:1.21",
      "actual": "nginx:1.21",
      "passed": true,
      "missing": false
    }
  ],
  "all_passed": true
}

When an assertion fails, the JSON shows the expected vs actual values for debugging:

"actual": 2,
"expected": 3,
"passed": false
