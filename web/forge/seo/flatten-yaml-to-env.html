Flatten YAML / JSON to Environment Variables

ConfigForge 1.0 11 formats --flatten / --unflatten

Kubernetes, Docker Compose, and most container platforms accept environment variables — but your team's config files are nested YAML. The typical solution is a fragile shell script with yq, jq, and awk pipelines that break every time the config structure changes.

ConfigForge --flatten converts any nested config to flat dot-notation key=value pairs in one command — ready for .env files, Kubernetes ConfigMaps, Docker Compose env_file, or export in shell scripts. --unflatten reverses the operation.

Quick Start

# Install
pip install devbench

# Flatten a nested YAML config to dot-notation key=value
devbench cf config.yaml --flatten

# Save as a .env file
devbench cf config.yaml --flatten --to env > .env

# Flatten JSON to YAML key=value (for Kubernetes ConfigMaps)
devbench cf app.json --flatten --to yaml

# Unflatten a .env back to nested YAML
devbench cf app.env --unflatten --to yaml

Example: Nested YAML to Flat Env Vars

# Input: config.yaml
database:
  host: postgres.internal
  port: 5432
  credentials:
    user: appuser
    password: secret123
redis:
  host: redis.internal
  port: 6379
  ttl: 3600
# devbench cf config.yaml --flatten
database.host=postgres.internal
database.port=5432
database.credentials.user=appuser
database.credentials.password=secret123
redis.host=redis.internal
redis.port=6379
redis.ttl=3600
# devbench cf config.yaml --flatten --to env
DATABASE_HOST=postgres.internal
DATABASE_PORT=5432
DATABASE_CREDENTIALS_USER=appuser
DATABASE_CREDENTIALS_PASSWORD=secret123
REDIS_HOST=redis.internal
REDIS_PORT=6379
REDIS_TTL=3600
Tip: Use --to env to get uppercase env var names (shell-compatible). Use --flatten without --to to get dot-notation (suitable for Spring Boot, ConfigMaps).

Real-World Use Cases

1. Kubernetes ConfigMap from Application Config

# Generate a Kubernetes ConfigMap from your app's YAML config
devbench cf app-config.yaml --flatten --to yaml \
  | kubectl create configmap app-config --from-env-file=/dev/stdin

# Or export as a ConfigMap manifest
cat > configmap.yaml << EOF
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
$(devbench cf app-config.yaml --flatten | sed 's/^/  /')
EOF

2. Docker Compose env_file Generation

# Generate .env from docker-compose.dev.yaml environment section
devbench cf docker-compose.yaml --get services.app.environment \
  | devbench cf - --flatten --to env > .env

# Load into the current shell session
eval "$(devbench cf config.yaml --flatten --to env | sed 's/^/export /')"

3. Spring Boot: YAML to .properties

# Spring Boot accepts both application.yaml and application.properties
# --flatten converts nested YAML to the flat .properties format directly
devbench cf src/main/resources/application.yaml --flatten --to properties \
  > src/main/resources/application.properties

# Input YAML:                  Output .properties:
# spring:                      spring.datasource.url=jdbc:postgresql://...
#   datasource:                spring.datasource.username=appuser
#     url: jdbc:...            server.port=8080
#     username: appuser
# server:
#   port: 8080

4. CI/CD: Inject Config as Environment Variables

# GitHub Actions: load config vars into the step environment
- name: Load config
  run: |
    devbench cf config/staging.yaml --flatten --to env >> $GITHUB_ENV

# Now all nested config keys are available as env vars in subsequent steps:
# DATABASE_HOST, REDIS_PORT, etc.

5. Terraform: Variables File from YAML

# Convert team's YAML config to Terraform .tfvars format
devbench cf infra-config.yaml --flatten --to toml > terraform.tfvars

# Or extract just the infrastructure section
devbench cf infra-config.yaml --get infrastructure --flatten --to env \
  > .terraform.env

6. Unflatten: Restore Nested Structure

# You have a flat .env — convert back to nested YAML for readability
devbench cf production.env --unflatten --to yaml

# Input .env:               Output YAML:
# DATABASE_HOST=db.host     database:
# DATABASE_PORT=5432          host: db.host
# REDIS_HOST=cache.host       port: 5432
# REDIS_PORT=6379           redis:
#                             host: cache.host
#                             port: 6379

Output Formats

--flatten (default: dot-notation)

database.host=postgres.internal
database.port=5432
database.credentials.user=appuser

--flatten --to env (uppercase env vars)

DATABASE_HOST=postgres.internal
DATABASE_PORT=5432
DATABASE_CREDENTIALS_USER=appuser

--flatten --to yaml (flat YAML — for ConfigMaps)

database.host: postgres.internal
database.port: 5432
database.credentials.user: appuser

Flags

FlagDescription
--flattenFlatten nested config to dot-notation key=value pairs
--unflattenRestore nested structure from flat dot-notation input
--to FORMATOutput format: env (uppercase), yaml, json, toml, properties
--get PATHExtract a subsection before flattening
--in-placeWrite result back to the input file
--batchFlatten all files matching a glob

How It Compares to Alternatives

Tool Input formats Flatten Unflatten TOML output Batch
ConfigForge --flatten 11 formats ✓ dot-notation ✓ --unflatten ✓ ✓
yq + jq pipeline YAML / JSON only ✓ (complex expr) ✗ ✗ ✗
python-dotenv .env only ✗ ✗ ✗ ✗
dasel 5 formats ✗ no flatten mode ✗ ✗ drops TOML comments ✗
yq flattening requires a complex expression:
yq eval '. | to_entries | map(select(.value | type == "string")) | from_entries' config.yaml
This only works one level deep and silently drops nested keys. ConfigForge handles arbitrary nesting depth and round-trips correctly with --unflatten.

Shell Script Examples

# Export all config vars into the current shell
source <(devbench cf config.yaml --flatten --to env | sed 's/^/export /')

# Generate a Kubernetes Secret from config values
kubectl create secret generic app-secrets \
  $(devbench cf secrets.yaml --flatten --to env \
    | sed 's/\(.*\)=\(.*\)/--from-literal=\1=\2/')

# Compare flattened configs to find divergence
devbench cf staging.yaml --flatten > /tmp/staging.flat
devbench cf production.yaml --flatten > /tmp/production.flat
diff /tmp/staging.flat /tmp/production.flat

# Or use ConfigForge --diff for structured output:
devbench cf staging.yaml --diff production.yaml

# Batch flatten all YAML configs in a directory
devbench cf 'configs/*.yaml' --batch --flatten --to env --output-dir ./env-files/

GitHub Actions Integration

name: Deploy with Config
on: [push]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install devbench

      - name: Load environment config
        run: |
          # Flatten staging config to GitHub Actions env
          devbench cf config/staging.yaml --flatten --to env >> $GITHUB_ENV

      - name: Deploy
        run: |
          # Config vars are now available: $DATABASE_HOST, $REDIS_PORT, etc.
          echo "Deploying to $DATABASE_HOST"
