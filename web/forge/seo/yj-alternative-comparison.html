ConfigForge vs yj: The Multi-Format yj Alternative That Keeps Your Comments


If you've landed here looking for a yj alternative, you're probably already using yj (sclevine/yj) to shuffle between JSON, YAML, TOML, and HCL on the command line. It's a tidy little Go binary that does exactly that — four formats, stream in, stream out. ConfigForge (the cf command in the open-source devbench toolkit) covers the same ground and then some: nine formats, and it's the only config converter that preserves comments through round-trip conversions. Here's the head-to-head.

Format Support: 9 vs 4


yj handles the four "structured config" formats that overlap most: JSON, YAML, TOML, and HCL (HCL is read-only — yj converts from HCL but not back to it). That's a genuinely useful set, especially if your work centers on Terraform and Kubernetes manifests.


ConfigForge is built for the messier reality of a real repository, where alongside those you'll also find a config.ini, a .env file, an exported data.csv, an XML manifest, and a Spring Boot application.properties. ConfigForge reads and writes all nine formats bidirectionally, including HCL in both directions, so any-to-any conversion works without reaching for a second tool:



# The yj classics — same one-liners, plus TOML and HCL both ways
devbench cf convert config.yaml config.json
devbench cf convert pyproject.toml settings.yaml
devbench cf convert main.tf main.json     # HCL → JSON
devbench cf convert main.json main.tf     # JSON → HCL (yj can't do this)

# The formats yj doesn't cover at all
devbench cf convert legacy.ini modern.yaml
devbench cf convert records.csv records.json
devbench cf convert manifest.xml manifest.yaml
devbench cf convert application.properties application.yaml

If your world is exactly JSON/YAML/TOML/HCL, yj is a fine, lean choice. The moment a CSV export, an INI file, or a .properties file enters the picture, you'd need to bolt on more tools — or just use ConfigForge for the whole matrix.


Comment Preservation — the Decisive Difference


This is the one that matters most day to day. Configuration files are documentation. The comment explaining why a timeout is 30 seconds, or why a feature flag is disabled, is often more valuable than the value itself. yj parses your config into a generic data structure and re-emits it — which means comments are dropped on the floor. Convert a commented config.yaml to TOML and back with yj and every annotation is gone.


ConfigForge is the only config converter that preserves comments across conversions wherever the target format supports them — YAML ↔ TOML ↔ INI ↔ .env all carry their inline and block comments through. Your annotations survive the round trip, so config files stay reviewable and your teammates keep the context they need.


Batch Mode


yj is a single-stream filter: one document in on stdin, one document out on stdout. That's perfect for piping, but converting a whole directory means writing a find ... -exec loop and handling errors yourself. ConfigForge ships native batch support with glob patterns, recursion, and per-file error reporting:



# Convert every JSON file in a tree to YAML, in place
devbench cf convert --batch --recursive "**/*.json" --to yaml

Offline Operation


Both tools run entirely on your machine — worth stating plainly because so many "online converter" websites ask you to paste sensitive config (API keys, database URLs, secrets) into a browser. ConfigForge makes zero external API calls. No telemetry, no cloud round-trip, no account. yj is likewise a local binary, so both win here over web tools — but only one of them also keeps your comments.


Who Should Use Which?


Choose yj if: your world is strictly JSON, YAML, TOML, and HCL, you want the smallest possible single static binary, and you're building stdin/stdout pipelines where a lightweight filter is all you need.


Choose ConfigForge (devbench cf) if: you work across more than those four formats, you need XML, CSV, INI, or .properties support, you want HCL conversion in both directions, you care about preserving comments through conversions, and you want native batch/recursive directory processing in a private, fully offline workflow.


For most polyglot codebases — and for anyone who has ever lost a carefully written config comment to a converter — ConfigForge is the broader, more future-proof choice. At pip install devbench-cf, it's one command away.
