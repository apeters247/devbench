Redact Sensitive Config Values with --mask

Strip passwords, tokens, and API keys from any config format before logging, sharing, or committing. One flag works across all 11 formats ConfigForge supports. 

# Mask credentials from a Kubernetes Secret or deployment config
devbench cf k8s-secret.yaml --mask 'password|secret|token'

# Convert YAML to JSON while redacting sensitive keys
devbench cf prod.yaml --mask 'password|api_key|private_key' --to json

# Custom redaction string
devbench cf .env --mask '.*' --mask-value '[REDACTED]' --to yaml

# Machine-readable output with redaction count
devbench cf config.yaml --mask 'password|secret' --raw | jq .redacted_count

The Problem: Secrets in Config Logs

CI/CD pipelines often print configs for debugging: 

echo "Deployed with config:"
cat deploy.yaml  # ← leaks DB_PASSWORD, API_KEY, JWT_SECRET to build logs

Fixing this without a dedicated tool requires complex jq expressions that only work for JSON, yq filter chains that don't handle all formats, or custom Python/bash scripts that you have to maintain.

ConfigForge's --mask PATTERN solves it in one flag across all 11 config formats: YAML, JSON, TOML, INI, .env, XML, CSV, HCL, .properties, plist, JSONC.

How It Works

--mask PATTERN takes a regex pattern and replaces the value of any key whose name matches the pattern. The replacement string defaults to ***REDACTED*** and can be customized with --mask-value.

  • Case-insensitive — --mask 'password' matches Password, PASSWORD, db_password
  • Recursive — walks all nested dicts and lists
  • Regex patterns — use alternation (password|secret|token), wildcards (.*key), anchors
  • Non-destructive — outputs a new config, never modifies the original file
  • Format-preserving — output stays in the same format unless you add --to
# Input (prod.yaml)
database:
  host: db.internal
  password: "sUp3rS3cr3t!"
  port: 5432
api:
  url: https://api.example.com
  secret_key: "aBcDeF123456"
  timeout: 30

# Output: devbench cf prod.yaml --mask 'password|secret'
database:
  host: db.internal
  password: '***REDACTED***'
  port: 5432
api:
  url: https://api.example.com
  secret_key: '***REDACTED***'
  timeout: 30

Real-World Use Cases

1. CI/CD Pipeline Logging

Log your merged config after templating without exposing credentials:

# GitHub Actions step
- name: Log deployment config
  run: devbench cf ${{ env.CONFIG_FILE }} --mask 'password|secret|token|key' | tee /tmp/deploy-log.yaml

2. Creating Sanitized Documentation Configs

Generate example configs from real configs with secrets stripped:

# Produce docs/config-example.yaml from prod config
devbench cf config/production.yaml --mask 'password|secret|token|key|credential' \
    --mask-value '<your-value-here>' \
    --to yaml > docs/config-example.yaml

3. Bug Report Sharing

Share configs with support or teammates without exposing production secrets:

devbench cf broken-config.yaml --mask '(?i)(pass|secret|key|token|auth|credential)' | pbcopy

4. Audit Log Export

Export masked configs to an audit log store:

devbench cf /etc/app/config.yaml --mask 'password|secret|api_key' --raw \
  | jq '{file: .file, redacted: .redacted_count, config: .output}' \
  >> /var/log/config-audit.jsonl

5. Kubernetes Secret Sanitization

Kubernetes Secrets have base64-encoded values under data.* keys — mask them before printing:

kubectl get secret my-secret -o yaml \
  | devbench cf - --from yaml --mask 'data' --to yaml

6. .env File Sharing

Share your .env structure with a new team member without exposing values:

devbench cf .env --mask '.*' --mask-value '<fill-in>' --to env > .env.example

The jq equivalent for masking nested password keys looks like this: 

# jq — JSON only, complex, hard to maintain
jq 'walk(if type == "object" then with_entries(if (.key | test("password|secret|token"; "i")) then .value = "***REDACTED***" else . end) else . end)'

devbench cf config.yaml --mask 'password|secret|token'
