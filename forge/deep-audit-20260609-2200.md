## 2. 🔴 Critical Issues (2)

### 🔴 CRITICAL-1: Predictable default HMAC secret — STILL UNFIXED
**File:** `web/license.py:82,494-517`
**Status:** Persistent since first audit; 3 cycles without fix.
**Impact:** Anyone knowing the hostname can forge valid license keys for the production license server. `_default_secret()` falls back to `sha256(hostname + machine + "ConfigForge-v1")` when `DEVBENCH_LICENSE_SECRET` env var is absent.
**Fix:** Delete `_default_secret()`. Replace with a hard crash: if `DEVBENCH_LICENSE_SECRET` is not set, raise a clear error message directing the operator to set it.

### 🔴 CRITICAL-2 (was HIGH-1): FIPS MD5 crash — STILL UNFIXED, upgraded severity
**File:** `core/tools.py:312`
**Status:** Previously HIGH, upgrading to CRITICAL for distribution: enterprise/government Linux with FIPS mode crashes on `hashlib.md5()` even with `usedforsecurity=False`. The PyPI package targets enterprises who are most likely to run FIPS.
**Impact:** Any `devbench hash` or `devbench detect` call on a FIPS-enabled system raises `ValueError`.
**Fix:**
```python
try:
    return hashlib.md5(data, usedforsecurity=False).hexdigest()
except ValueError:
    return hashlib.sha256(data).hexdigest()
```

## 3. 🟡 High Issues (3)

### 🟡 HIGH-1: License server rate limiter is NOT thread-safe — STILL UNFIXED
**File:** `web/license_server.py:93-107`
**Status:** Persistent from previous audit (was HIGH-NEW2).
**Detail:** `_check_rate()` uses module-level dict `_rates: dict[str, list[float]] = {}` with no `threading.Lock`. Under `ThreadingHTTPServer`, concurrent requests can race: two threads read the same list, both pass the rate check, and append independently — clobbering each other. `web/api.py`'s `RateLimiter` class does this correctly with a `threading.Lock`.
**Fix:** Either import `RateLimiter` from `web/api.py` or add `threading.Lock()` to `_check_rate()`.

### 🟡 HIGH-2: Uncommitted working tree — 45 tests + 5 features not in git
**File:** `core/cli.py`, `core/configforge.py`, `tests/test_core.py`, `web/forge/seo/minify-json-cli.html`
**Status:** NEW this cycle.
**Detail:** The following features are implemented and tested but NOT committed: `--sort-keys-reverse`, `--compact`, `--template`, `--get --default`, `--select FIELD=VALUE`, plus `minify-json-cli.html` SEO page. A crash, OOM, or accidental checkout would lose this work. The last commit (157d333) records only 1087 tests; the live tree has 1132.
**Fix:** Commit immediately: `git add core/cli.py core/configforge.py tests/test_core.py web/forge/seo/minify-json-cli.html && git commit`.

### 🟡 HIGH-3: No concurrent/stress testing for any threaded server
**File:** `web/api.py`, `web/serve.py`, `web/license_server.py`
**Status:** Persistent from previous audit.
**Detail:** All three servers use `ThreadingHTTPServer` but zero concurrency tests exist. Race conditions in rate limiters (see HIGH-1), SQLite `database is locked` under high concurrency (`web/license.py` opens a new `sqlite3.connect()` per call), and CORS handling are undetected. This is the highest-risk gap before the commercial launch.

## 4. 🟢 Medium Issues (9)

### 🟢 MEDIUM-1: TOCTOU race in test port selection — STILL UNFIXED
**File:** `tests/test_license.py:409-412`
**Status:** Persistent from first audit; 4 cycles without fix.
**Detail:** Open socket on port 0, get port, close socket, start server. Between close and start another process can steal the port. Affects 4 `_start_server` instances.

### 🟢 MEDIUM-2: `ToolResult` Pydantic model is dead code — STILL UNFIXED

**Status:** Persistent.
**Detail:** 70-line `ToolResult` model with `to_swiftui()`, `error_result()`, `success_result()` is never instantiated. All tool output goes through `_ok()`/`_err()` in `tools.py`. Either adopt it or delete it.

### 🟢 MEDIUM-3: `test_edge_cases.py` monolithic — STILL GROWING
**File:** `tests/test_edge_cases.py`
**Status:** Still unfixed; likely larger than last audit's 2123 lines.
**Detail:** Previous audit noted 2123 lines. Not remediated. Needs splitting per format.

### 🟢 MEDIUM-4: configforge.py standalone CLI is 5 features behind cli.py — NEW
**File:** `core/configforge.py:2931` (`main()` function at the bottom)
**Status:** NEW this cycle.
**Detail:** The `configforge` entry point's `main()` (3260 lines into the file, the legacy CLI) is missing: `--sort-keys-reverse`, `--compact`, `--template`, `--get --default`, and `--select`. A user who invokes `configforge` directly (or uses it as a library via `configforge.main()`) gets a different feature set than `devbench cf`. This gap will grow each release cycle if not addressed.
**Fix:** Either delete `configforge.main()` (and remove the `configforge` console_script entry point), or auto-generate it from the same argparse spec as `cli.py`.

### 🟢 MEDIUM-5: Two duplicate rate limiter implementations
**File:** `web/api.py:96-155` vs `web/license_server.py:93-107`
**Status:** Persistent; one is thread-safe, one is not.

### 🟢 MEDIUM-6: `sqlite3.connect()` per-call in license server
**File:** `web/license.py` (`_count_activations()` and related)
**Status:** Persistent from previous audit.
**Detail:** Every license activation/verify call opens a new SQLite connection. Under moderate concurrency (10+ simultaneous API requests) this produces `sqlite3.OperationalError: database is locked`. Use a connection pool or a single persistent connection with `check_same_thread=False`.

### 🟢 MEDIUM-7: `batch_convert_stream()` — fixed test gap but streaming output path conflicts
**File:** `core/configforge.py:2700-2763`
**Status:** Test gap FIXED (4 tests added). New finding: when `show_progress=True` (the default), the streaming function prints progress to `stdout` via `print()`. CLI callers in `_run_cf_batch` correctly override this with `\r` progress display. But library callers who call `batch_convert_stream()` directly without `show_progress=False` will get progress output mixed into their stdout. The streaming API has no clean way to separate progress from results.
**Severity:** MEDIUM — only affects library users, not CLI users.

### 🟢 MEDIUM-8: Missing SEO pages for 3 new features
**Status:** NEW this cycle.
**Detail:** Pattern of creating SEO landing pages for each flag was consistent through `--path-exists`/`--shell-export`. The new features are missing pages:
- `--get --default`: no page (jq's `// empty` alternative — high search value)
- `--sort-keys-reverse`: no page (yq#2390 pain point)
- `--template`: no page (high-value: "render template from config")
`--select` has `filter-yaml-array-by-field.html` ✅. `--compact` has `minify-json-cli.html` ✅ (untracked).

### 🟢 MEDIUM-9: No request timeouts on any HTTP server
**File:** All 3 web servers
**Status:** Persistent.
