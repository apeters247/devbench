## The Gap

Users want a **reliable, safe, offline config file tool** that handles their actual workflows. The codebase has 679 tests and covers 7 formats, but 4 fundamental architectural problems prevent it from being the tool users imagine.

## Problem 1: Monolithic Converter, Not a Pipeline

`configforge.py` is a single 1119-line file with one function per format. Every format's parser, serializer, comment handler, and type inference is baked into the same module. The user complaints (#2 "offline converter", #3 "unified CLI") expect ONE tool — but configforge.py is the *engine*, not the ecosystem.

**What's fundamentally wrong:** Adding a new format (HCL, .properties, .plist) requires editing the core file, re-testing everything, and risking regressions. The architecture has no plugin system, no format registry, no adapter interface.

**Users actually want:** `forge add-format hcl` or dropping a `formats/hcl.py` file. The comment preservation, type inference, and round-trip logic should be reusable decorators, not copy-pasted for each format.

## Problem 2: Comment Preservation is a Hack, Not a Feature

The comment system (`_COMMENT_CACHE`, `_extract_yaml_comments`, `_reinsert_yaml_comments`) treats comments as external strings extracted before conversion and reinserted after. This works for simple cases but:

- Can't handle YAML→TOML→YAML with comments (comments are lost in TOML intermediate)
- Duplicate key names at different depths cause comment misplacement
- No error if a comment can't be reinserted (silent drop)
- Line-index shifting on insertion is O(n²) with every insertion shifting subsequent offsets

**Users actually want:** Comments as first-class data that survive ANY intermediate format. The correct approach is attaching comments to the AST/data model as metadata, not string manipulation. JSON comments in `//` form could be a bridge.

## Problem 3: Type Inference Is Fragile

`_infer_type` (line 380) uses regex + try/except. It runs on every INI value but doesn't run on values parsed from JSON, YAML, or TOML (which already have native types). This means round-trips can change types:
- JSON `{"x": "1"}` → INI → JSON: `"1"` stays string ✓
- JSON `{"x": 1}` → INI → JSON: `1` stays int ✓
- JSON `{"x": "true"}` → INI → JSON: becomes boolean `true` ✗ (string→bool coercion on the return leg)

**Users actually want:** Predictable type preservation. The CLI should have `--string-only` and `--infer-types` flags, and type inference should be opt-in per-format, not automatic.

## Problem 4: Error Handling Surface

`convert()` catches all exceptions and returns `{"success": False, "error": str(e)}`. This is good for UX but means the CLI never sees unhandled errors — including bugs that should be surfaced. Calls to `_to_toml` and `_from_dict_to_xml` can raise `ValueError` for genuinely unrecoverable cases, but batch mode treats these as per-file errors and continues. No structured error codes, no warning for data loss.

**Users actually want:** Warn-level diagnostics (e.g., "Comment lost from line 42"), not silent data dropping. Error codes, not string matching on error messages.

### Phase 2 (2 weeks): Comment-Aware Data Model
Introduce `ConfigData` dataclass: `class ConfigData(dict): comments: dict[str, list[str]]` where keys map to comment lists. Parsers populate it, serializers read it. Eliminate `_COMMENT_CACHE` and the extract/reinsert pattern. All 679+ tests must pass.
