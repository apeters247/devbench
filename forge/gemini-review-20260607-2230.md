### Critical / High-Severity Findings

1. **GitHub Actions CI workflow has broken `uses:` references** — The review claims `.github/workflows/ci.yml` references local script paths (e.g., `@../herbalist/...`). **This is partially a false positive** — the actual `uses:` lines in the diff correctly reference `actions/checkout@v4`, `actions/setup-python@v5`, `actions/upload-artifact@v4`, `pypa/gh-action-pypi-publish@release/v1`. However, Gemini's claim may point at a different file context not fully shown in the truncated diff. ✅ **Verified: the published CI workflows are correct.**

2. **`tomllib.loads()` requires Python 3.11+** — The test `test_convert_json_to_toml` (test_configforge.py:38) now uses `tomllib.loads()`. The project `pyproject.toml` specifies `requires-python = ">=3.10"`. This test will **fail on Python 3.10**. **Action item**: bump `requires-python` to `>=3.11` or use `tomli` backport.

3. **`_sort_json_keys` sorts list elements** — Referenced from prior Gemini review as an existing bug. Gemini flagged that `_sort_json_keys` sorts objects within arrays, potentially destroying semantic ordering. Worth verifying in `core/tools.py`.

4. **Release scripts mask failures with `|| true`** — `scripts/publish-pypi.sh` and `scripts/release.sh` use `|| true` on `git commit` and `|| warn` on `git push`. A silent failure here could create partial releases (PyPI updated but GitHub not tagged). **Action item**: use `git diff --cached --exit-code || git commit` for conditional commits; hard-fail on push during release.

### Medium-Severity Findings

5. **`test_decode_returns_metadata` fails** — `web/license.py` `decode()` method does not return a `"key"` field in its dict, but the test `test_license.py:149` expects `info["key"] == key`. This is a **real test failure** (confirmed: 1 failed, 573 passed). Either `decode()` should include the original key, or the test expectation is wrong.

6. **`comment_loss_warning` stderr output not tested** — `core/cli.py` prints the warning to `sys.stderr`, but the new tests in `test_edge_cases.py` only assert the result dict field. No `capsys` test verifies the CLI actually emits the warning to the user. Easy regression vector.

7. **macOS CI `.app` bundle copy is brittle** — Uses `cp -R ... 2>/dev/null || ...` which suppresses errors silently. If Xcode build output varies, the artifact could be missing without a clear CI failure.

8. **Homebrew formula `sha256` is a placeholder** (Formula/devbench.rb:14) and `revision 1` is unusual for a first formula (should be `revision 0`). Manual release step that can be forgotten.

### Low-Severity Findings
