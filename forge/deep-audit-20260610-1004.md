## 4. 🟢 Medium Issues (6)

### 🟢 MEDIUM-1: TOCTOU race in test port selection — STILL UNFIXED
**File:** `tests/test_license.py`
**Status:** Persistent from first audit (6 cycles without fix).
**Detail:** `_start_server` opens a socket on port 0, reads the bound port, closes the socket, then starts the server on that port. A concurrent process can steal the port between close and start. Use `SO_REUSEPORT` or keep the socket open while the server binds.

### 🟢 MEDIUM-2: `ToolResult` Pydantic model is dead code — STILL UNFIXED
**File:** `core/models.py` (entire file)
**Status:** Persistent; 7 cycles without fix.
**Detail:** 70-line `ToolResult` model with `to_swiftui()`, `error_result()`, `success_result()` is never instantiated. All tool output goes through `_ok()`/`_err()` in `tools.py`. Either adopt it or delete it.

### 🟢 MEDIUM-3: `test_edge_cases.py` monolithic — STILL GROWING
**File:** `tests/test_edge_cases.py`
**Status:** Persistent. 2267 lines (previously estimated >3000 — estimate was wrong; actual count is lower).
**Detail:** 249 test functions in one file covering all formats. Now growing again with new feature tests. Should be split per format or per feature area to improve discoverability and CI parallelism.

### 🟢 MEDIUM-4: `configforge.main()` is now dead code — PARTIALLY FIXED / NEW FORM
**File:** `core/configforge.py:3187–3321`
**Status:** Previously MEDIUM-4 (growing feature gap). Entry point removed from `pyproject.toml` ✅. Function body not deleted.
**Detail:** The `configforge` CLI entry point is gone — no user can invoke `configforge.main()` from an installed package. But the function body (134 lines) still exists. The stdin-path inside the stub calls `convert()` directly with only 6 options wired (sort-keys, yaml12, template-safe, block-scalars, no-comments, compact), missing 35+ flags. This is unreachable dead code but is misleading to future maintainers.
**Fix:** `git rm` the `configforge.main()` function body and its `from core.cli import main as _cli_main` import.

### 🟢 MEDIUM-5: `batch_convert_stream()` progress output contaminates stdout — STILL UNFIXED
**File:** `core/configforge.py:2700-2763`
**Status:** Persistent.
**Detail:** When `show_progress=True` (default), streaming function prints progress to stdout via `print()`. Library callers who call `batch_convert_stream()` directly without `show_progress=False` get progress output mixed into their stdout.
**Fix:** Change default to `show_progress=False` or route progress to stderr.

### 🟢 MEDIUM-6: No request timeouts on any HTTP server — STILL UNFIXED
**File:** `web/api.py`, `web/serve.py`, `web/license_server.py`
**Status:** Persistent from previous audit.
**Detail:** Slow clients can hold connections indefinitely. No `socket.settimeout()` on incoming connections. Under concurrent load (which is now untested — see HIGH-1), this becomes a resource exhaustion vector.
