# Migrate from Ansible INI Inventory to YAML: A Complete Conversion Guide

If you're maintaining Ansible inventory in INI format and considering moving to YAML, you're not alone. Modern Ansible best practices recommend YAML inventory for better structure, nested group support, and variable inheritance. But manually converting hundreds of lines of INI to YAML is error-prone and tedious.

This guide shows how to **convert Ansible INI to YAML** (and back) using **ConfigForge** (`devbench cf`) — a free, offline CLI that handles 9 config formats. Your hostnames, SSH users, and connection strings stay on your machine — nothing is uploaded anywhere.

## Why Migrate from INI to YAML?

| INI Inventory | YAML Inventory |
|---|---|
| Flat group definitions | Nested group hierarchy |
| Limited to `:children` and `:vars` | Full YAML dict/list structure |
| No comments per-host | Inline comments on every entry |
| Hard to version-control diff | Clean, readable YAML diffs |
| Parsing-dependent type handling | Native type inference |

## Basic INI to YAML Conversion

Start simple — convert a single inventory file:

```bash
devbench cf --to yaml --from ini < production.ini
```

ConfigForge detects the INI sections (`[group]`), children (`[group:children]`), and variables (`[group:vars]`), and produces clean, structured YAML.

## Handling Group Inheritance

INI uses `:children` syntax for group membership:

```ini
[web:children]
app_servers
cache_servers

[app_servers]
web01 ansible_host=10.0.1.10
web02 ansible_host=10.0.1.11

[cache_servers]
redis01 ansible_host=10.0.2.10
```

ConfigForge converts this to nested YAML:

```yaml
web:
  children:
    app_servers:
      hosts:
        web01:
          ansible_host: 10.0.1.10
        web02:
          ansible_host: 10.0.1.11
    cache_servers:
      hosts:
        redis01:
          ansible_host: 10.0.2.10
```

## Converting Group Variables

INI group variables become clean YAML dicts:

```ini
[db:vars]
ansible_user=deploy
ansible_port=22
postgres_version=15
debug=false
max_connections=100
```

ConfigForge's type inference converts the values so booleans stay booleans and numbers stay numbers:

```yaml
db:
  vars:
    ansible_user: deploy
    ansible_port: 22
    postgres_version: 15
    debug: false
    max_connections: 100
```

Note `debug=false` becomes YAML boolean `false` (not the string `"false"`), and `max_connections=100` becomes integer `100`.

## Converting Ansible Vars YAML to JSON

After migration, you may need to feed Ansible variable files to JSON-only tools:

```bash
devbench cf --to json < group_vars/all.yml
```

This converts your Ansible vars from YAML to JSON, preserving the nested dict structure.

## Batch Converting an Entire Inventory Directory

For a full migration with multiple inventory files:

```bash
devbench cf convert "inventory/*.ini" --to yaml --from ini --batch
```

Or recursively for nested inventory structures:

```bash
devbench cf convert "inventory/**/*.ini" --to yaml --from ini --batch
```

## Security: Why Your Ansible Inventory Should Never Go Online

Ansible inventory files contain:
- Server IP addresses and internal DNS hostnames
- SSH user accounts and connection ports
- Connection strings with database credentials
- Environment-specific topology patterns

**Never paste Ansible inventory into an online converter.** ConfigForge runs entirely on your machine — no uploads, no API calls, no telemetry.
