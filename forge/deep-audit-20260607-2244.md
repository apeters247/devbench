## 2. 🔴 Critical Issues (2)

### 🔴 CRITICAL-1: Predictable default HMAC secret — STILL UNFIXED
**File:** `web/license.py:82,494-517`
**Previous audit:** CRITICAL-1 (unchanged)
**Impact:** Anyone who knows the hostname (visible in server headers, DNS) can forge valid license keys.
**Detail:** `_default_secret()` at line 494 still falls back to `sha256(hostname + machine + "ConfigForge-v1")` when `DEVBENCH_LICENSE_SECRET` is not set AND `DEVBENCH_DEV` is not `=1`. The code path is: `LicenseManager.__init__` calls `_default_secret()` which raises ValueError if neither env var is set. But many test and dev scenarios hit the ValueError and crash rather than silently degrading. The real issue is that this code *exists* at all — it should be deleted or replaced with a hard error that tells the operator to set the env var.

### 🔴 CRITICAL-NEW1: HCL venv path hardcoded to `/tmp/devbench_venv`
**File:** `core/configforge.py:49`
**Impact:** `pip install devbench` on any system that doesn't have `/tmp/devbench_venv` will have `HAS_HCL = False`. The project documents 9 formats, but HCL silently degrades on clean installs.
**Detail:**
```python
for _hcl_sp in _glob.glob("/tmp/devbench_venv/lib/python*/site-packages"):
```
This is the builder's work VM path. It will not exist on user machines. When HCL ops are attempted, `HAS_HCL` is `False` and all HCL conversions fail with "HCL support requires python-hcl2 library." Either:
- Ship `python-hcl2` as a project dependency in `pyproject.toml`
- Or remove HCL from `SUPPORTED_FORMATS` until it can be genuinely bundled

## 3. 🟡 High Issues (4)

### 🟡 HIGH-1: FIPS crash — `hashlib.md5()` on FIPS systems — STILL UNFIXED
**File:** `core/tools.py:312`
**Previous audit:** HIGH-1 (unchanged)
**Impact:** Crashes on enterprise/government Linux with FIPS mode. `hashlib.md5(data, usedforsecurity=False)` still raises `ValueError` on FIPS-enforced systems. The `usedforsecurity=False` flag works on *most* systems but NOT on FIPS-enabled kernels where MD5 is completely disabled at the OpenSSL level.
**Fix:** Wrap in try/except with fallback to SHA-256, or detect FIPS mode and skip MD5.

### 🟡 HIGH-NEW2: License server rate limiter is NOT thread-safe
**File:** `web/license_server.py:93-107`
**Impact:** The server uses `ThreadingHTTPServer` (threaded), but the `_check_rate()` function uses a plain module-level dict `_rates: dict[str, list[float]] = {}` with no `threading.Lock`. Two concurrent requests from the same IP can race:
```python
_rates[ip] = [t for t in _rates[ip] if now - t < _RATE_WINDOW]  # RACE: two threads read same list
```
One thread's prune can clobber the other's append, or both threads can concatenate and both pass the limit check. This rate limiter is ineffective under concurrency.
**Fix:** Add `threading.Lock()` like `web/api.py:RateLimiter` already does. Or better: reuse the `RateLimiter` class from `api.py` instead of duplicating the logic.

### 🟡 HIGH-NEW3: `batch_convert_stream()` completely untested — 0 tests
**File:** `core/configforge.py:1806-1878`, `tests/` (no coverage)
**Impact:** The streaming batch mode (used for 10K+ file workloads) has zero test coverage. Any bugs in `batch_convert_stream()` — the generator wrapper that handles progress bars, output directory creation, comment loss warnings, and summary dicts — are undetected. This is the most complex new feature without any test.
**Fix:** Add at minimum: (a) basic glob conversion test, (b) empty glob test, (c) mixed success/failure test, (d) output-dir creation test.

### 🟡 HIGH-NEW4: `ignore` must be listed in `pyproject.toml` or `test_extend_expiry_malformed_key` flake in CI
**File:** `tests/test_license.py:370-379`
**Detail:** The test iterates `("", "not-a-key", "CF.", "CF.onlytwo", "garbage.with.dots")` calling `lm.extend_expiry(bad, 86400)`. For the empty string `""`, the DB query runs `WHERE license_key = ?` with `""` — this succeeds on SQLite but may return different results by implementation. On CI with a fresh DB, this works, but if the DB had any prior insertions it could flake. Minor but worth noting.

## 4. 🟢 Medium Issues (8)

### 🟢 MEDIUM-1: TOCTOU race in license test port selection — STILL UNFIXED
**File:** `tests/test_license.py:409-412` (inside `_start_server` methods)
**Previous audit:** MEDIUM-1 (unchanged)
**Detail:** Opens a socket on port 0 (ephemeral), gets the port, closes the socket, then starts the server on that port. Between close and start, another process can steal the port. Still unfixed across all 4 `_start_server` instances in `TestServerEndpoints` and `TestGumroadWebhook`.

### 🟢 MEDIUM-2: `ToolResult` Pydantic model is completely unused
**File:** `core/models.py` (entire file), cross-referenced: `core/tools.py`, `core/detector.py`, `core/cli.py`
**Impact:** The `ToolResult` model (70 lines with `to_swiftui()`, `error_result()`, `success_result()` factories) is NEVER constructed anywhere. All tool functions produce raw dicts via `_ok()`/`_err()` helpers in `tools.py`. The model is dead code that creates maintenance burden.
**Count:** `grep -rn "ToolResult" core/` returns only the import in `detector.py` docstring — not a single instantiation.
**Fix:** Either start using it, or delete it. If kept, the CLI `_run_cf()`, `detect_and_run()`, and all tool functions should return `ToolResult` instances.

### 🟢 MEDIUM-3: `test_edge_cases.py` monolithic (2123 lines) — growing
**File:** `tests/test_edge_cases.py`
**Previous audit:** MEDIUM-11 (was 1899 lines, now 2123 lines)
**Detail:** The file has grown by 224 lines since the last audit (now 2123 lines, 239 test functions). This is the single largest test file. The previous recommendation to split per format has not been actioned. New sections 20-21 added for external-review regression tests.

### 🟢 MEDIUM-4: `serve.py` calls `ap.parse_args()` twice
**File:** `web/serve.py:526`
**Impact:** At the script entry point:
```python
raise SystemExit(run_server(ap.parse_args().port, ap.parse_args().host))
```
`ap.parse_args()` is called twice, re-parsing `sys.argv` each time. This is wasteful and fragile: if argparse ever gains side effects (e.g., consuming stdin), this could produce inconsistent results. Should be:
```python
args = ap.parse_args()
raise SystemExit(run_server(args.port, args.host))
```

### 🟢 MEDIUM-5: Two duplicate rate limiter implementations
**File:** `web/api.py:96-155` (class-based `RateLimiter` with lock) vs `web/license_server.py:93-107` (module-level dict, no lock)
**Impact:** Two implementations of the same algorithm, one thread-safe (api.py) and one not (license_server.py). The license_server version also duplicates code rather than importing from `api.py`. This violates DRY and the non-thread-safe version is a correctness bug.

### 🟢 MEDIUM-6: License server `<html>` variable conflicts with HTML content
**File:** `web/license_server.py:169-170`
**Detail:** The variable `download_html` stores a `Path` object, then the code checks `.exists()` and `.read_text()`. This is correct in current code but the variable name suggests it holds HTML when it holds a file path. Readability issue only.

### 🟢 MEDIUM-7: `test_swift_bridge_contract.py` doesn't test `cf` tool output format
**File:** `tests/test_swift_bridge_contract.py:255`
**Detail:** `test_all_subcommands_produce_json` only checks that `output` is a string — it does not verify that `cf` (ConfigForge) output is well-formed config. Also `DETECT_SAMPLES` at line 127-132 contains no `cf`/config-format samples (no YAML, TOML, INI). The Swift bridge test is incomplete for the most complex tool.

### 🟢 MEDIUM-8: No concurrent/stress testing for any threaded server
**Detail:** Three `ThreadingHTTPServer` instances (`api.py`, `serve.py`, `license_server.py`) with zero concurrency tests. Race conditions in rate limiters, DB access, file writes, and CORS handling are undetected. `web/license.py`'s `_count_activations()` opens/closes a new `sqlite3.connect()` per call — under high concurrency this will cause `database is locked` errors.
