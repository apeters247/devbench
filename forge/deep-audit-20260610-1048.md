## 4. High Issues (🟡)

### HIGH-1 — No concurrent-request tests for threaded HTTP servers
**File:** `web/api.py`, `web/serve.py`, `web/license_server.py`
**Status:** UNFIXED — 7th consecutive audit cycle.

All three servers use `ThreadingHTTPServer`, which spawns a new thread per request. There are no tests that fire 2+ simultaneous requests to verify:
- Shared state under concurrent writes (e.g. license server's in-memory `_licenses` dict)
- Race conditions in `web/api.py`'s stateless handler (likely safe, but untested)
- Thread-safety of `web/serve.py`'s static file serving path

**Risk:** A production deployment handling concurrent requests could corrupt license state or produce garbled responses.

## 5. Medium Issues (🟢)

### MEDIUM-1 — TOCTOU port race in test `_start_server`
**File:** `tests/test_serve.py`, `tests/test_api.py`
**Status:** UNFIXED.

Tests pick a free port with `socket.bind(('', 0))`, release it, then pass it to the server. Between release and bind, another process can grab the port. Causes flaky CI failures.

**Fix:** Use `SO_REUSEADDR` + pass `port=0` to the server and read back the assigned port from `server.server_address[1]`.

### MEDIUM-2 — Dead `ToolResult` model
**File:** `core/models.py:12`
**Status:** UNFIXED.

`ToolResult` (Pydantic-backed or stub) is defined but never imported by `core/cli.py`, `core/tools.py`, or any test. It is dead code.

**Fix:** Delete the class and its no-op stub (both branches of the `try/except` import).

### MEDIUM-3 — `detect_format()` rejects valid single-line YAML scalars
**File:** `core/configforge.py:1365-1391`
**Status:** UNFIXED.

A bare YAML scalar (`"hello"` or `42`) falls through all format checks and returns `"unknown"`. While unusual, YAML 1.1/1.2 allows bare scalars as documents.

**Impact:** Low in practice (most real configs are mappings/sequences).

### MEDIUM-4 — `configforge.main()` body is dead code (134 lines)
**File:** `core/configforge.py:3186-3320`
**Status:** UNFIXED.

`configforge.main()` exists and delegates to `cli.main()` for backward compat, but the `configforge` entry point was removed from `pyproject.toml` in a prior cycle. Nothing calls it. The entire 134-line function body is unreachable from any installed entry point.

**Fix:** Delete the function. If backward compat is needed, keep a one-liner stub that raises `NotImplementedError`.

### MEDIUM-5 — `batch_convert_stream()` defaults `show_progress=True`
**File:** `core/configforge.py:2912`, `core/configforge.py:2984`
**Status:** UNFIXED.

Both `batch_convert()` and `batch_convert_stream()` default `show_progress=True`. Library consumers (tests, programmatic callers) must explicitly pass `show_progress=False` to suppress noisy progress output to stdout. A library function should default to silent.

**Fix:** Change `options.pop("show_progress", True)` to `options.pop("show_progress", False)` in both functions and update the 3 tests that check for progress output with `show_progress=True`.

### MEDIUM-6 — `_COMMENT_CACHE` dict is populated but never read
**File:** `core/configforge.py:113`
**Status:** UNFIXED.

`_COMMENT_CACHE = {}` is defined at module level. No code path reads from it; comments are extracted fresh on every call.

**Fix:** Remove the declaration (zero behavioral impact, reduces confusion).

## 6. Low Issues (⚪)

### LOW-1 — `_extract_ini_comments` does not handle `!` as a standalone comment marker
**File:** `core/configforge.py:769-784`
**Status:** UNFIXED.

The `.properties` spec uses `!` as a comment prefix (same as `#`). `_extract_ini_comments` handles `!` in the loop only when checking `stripped.startswith(("#", ";"))` — it skips `!`-only comments for INI files. Minor edge case; unlikely to affect real users.

### LOW-2 — `_infer_type()` treats `yes`/`no`/`on`/`off` as booleans (YAML 1.1 behavior)
**File:** `core/configforge.py:1491`
**Status:** UNFIXED (intentional, but not documented in help).

INI/ENV parsers call `_infer_type()` which converts `yes/no/on/off` to Python booleans — YAML 1.1 semantics. A `--yaml12` flag controls YAML parsing but does not affect INI/ENV value inference. Users converting INI → TOML/JSON with `enabled=no` will get `false` in the output, which may or may not be desired.

### LOW-3 — `_count_delims_outside_quotes` treats single and double quotes identically
**File:** `core/configforge.py:93-109`
**Status:** UNFIXED.

The CSV delimiter counter toggles `in_quotes` on any quote character, so a line like `a,'b"c',d` is miscounted (the `"` inside the single-quoted string toggles the quote flag). Affects CSV auto-detection for files mixing quote styles. Extremely rare in practice.

### LOW-4 — `_strip_json5` emits `text[i + 1:key_start]` raw
**File:** `core/configforge.py:1269`
**Status:** UNFIXED.

When quoting an unquoted JSON5 key, the code emits `result.append(text[i + 1:key_start])` for whitespace between `{`/`,` and the key. If that span contains a newline, the output is valid JSON but the line counting used in error messages would be off. Not a correctness bug, but a potential diagnostic issue.

### LOW-5 — `configforge.main()` has a typo in env var name
**File:** `core/configforge.py:3183`
**Status:** UNFIXED (also covered by MEDIUM-4 — delete the function).

`DEVEBENCH_NO_TELEMETRY` (missing 'n') is checked alongside `DEVBENCH_NO_TELEMETRY`. This is vestigial typo-tolerance code that would survive if the function is ever reinstated.

## 8. Architecture Notes

- **`core/configforge.py` is 3,320 lines** — the largest file by far. Format-specific parse/serialize logic could be split into `core/formats/yaml.py`, `core/formats/toml.py`, etc. Not urgent at 1.0, but worth doing before adding format #12.
- **`core/cli.py` is 4,663 lines** — the `_build_parser()` function alone is ~450 lines. Splitting per feature domain (CRUD ops, list ops, server ops) would improve maintainability.
- **Three separate HTTP servers** (`web/api.py`, `web/serve.py`, `web/license_server.py`) all re-implement request routing with raw `BaseHTTPRequestHandler`. Consolidating into one server with route registration would eliminate duplicated logic.
- **No structured logging** — all three servers use `print()` for both progress and errors. `logging.getLogger()` would allow log-level control in production.
