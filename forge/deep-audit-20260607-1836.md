## 1. Executive Summary

**23 bugs found (3 critical, 8 high, 9 medium, 3 low), 40+ tech debt items, 50+ test gaps, 20 architecture issues.** The codebase is functional and well-tested (551/560 passing, 0 failures) but carries significant technical debt from rapid iteration. The deepest problems are in the **license server** (predictable HMAC secrets, Stripe webhook bypass, duplicate license generation on renewal), **configforge.py** (2084-line monolith), and **test quality** (pervasive `is not None` weak assertions, sleep-based server waits, hardcoded magic numbers).

**Test suite health is misleading** — many tests pass without verifying correctness (asserting `r is not None` rather than checking output content). Real bug coverage is lower than the green suite suggests.

## 2. 🔴 Critical Issues (3)

### 🔴 CRITICAL-1: Predictable default HMAC secret for license keys
**File:** `web/license.py:430-441`
**Impact:** Anyone who knows the hostname (visible in server headers, DNS, etc.) can forge valid license keys.
**Detail:** `_default_secret()` falls back to `sha256(hostname + machine + "ConfigForge-v1")`. The comment says "fine for dev/test" but this code path is active whenever `DEVBENCH_LICENSE_SECRET` is not set, which is common in dev/deploy. This means all development and demo deployments produce forgeable licenses.

### 🔴 CRITICAL-2: Stripe webhook signatures never verified by default
**File:** `web/license_server.py:335-336`
**Impact:** Anyone who deploys without explicitly setting `DEVBENCH_DEV=0` (default is `"1"`) accepts unsigned Stripe events — attackers can forge webhook calls and generate free license keys.
**Detail:** `DEVBENCH_DEV` defaults to `"1"`, making `is_dev = True`. The signature verification block is entirely skipped when `is_dev` is True. Combined with 🔴-1, a default deployment has both forgeable license keys AND forgeable Stripe webhooks.

### 🔴 CRITICAL-3: `invoice.paid` generates duplicate license keys on renewal
**File:** `web/license_server.py:357-359`
**Impact:** Every billing renewal creates a duplicate, useless license key for the customer instead of extending the existing key.
**Detail:** When Stripe sends `invoice.paid`, `_process_checkout` is called, which generates a brand-new license key. The customer already has one from the original `checkout.session.completed`. The code should check for an existing key and extend it, or at minimum de-duplicate.

## 3. 🟡 High Issues (8)

### 🟡 HIGH-1: FIPS crash — `hashlib.md5()` on FIPS-enforced systems
**File:** `core/tools.py:316`
**Impact:** Crashes on any enterprise/government Linux that has FIPS mode enabled. `hashlib.md5()` raises `ValueError` because MD5 is disabled.
**Who should fix:** Devbench Build (owns tools.py)

### 🟡 HIGH-2: POST routing bug — query strings silently cause 404
**File:** `web/license_server.py:208`
**Impact:** POST requests with URL query parameters (e.g., `/license/activate?foo=bar`) always return 404 because `do_POST` uses `self.path` directly (which includes query string) instead of `urlparse(self.path).path` like `do_GET` does. All POST endpoints silently break when clients include URL parameters.

### 🟡 HIGH-3: `str.removeprefix()` requires Python 3.9+
**File:** `web/serve.py:412`
**Impact:** Crashes with `AttributeError` on Python 3.8 (still the default on Ubuntu 20.04 LTS, Debian 11, RHEL 8).
**Fix:** Use `rel = path[len("/demo/"):] if path.startswith("/demo/") else path` or a try/except fallback.

### 🟡 HIGH-4: Daemon cleanup thread silently dies on exception
**File:** `web/api.py:101-108`
**Impact:** If `cleanup()` ever raises an exception, the daemon thread terminates permanently and the rate limiter's `_hits` dict grows unbounded — no recovery mechanism, no alerting.

### 🟡 HIGH-5: Information disclosure — Python exception details leaked to API clients
**File:** `web/api.py:211,223`, `web/serve.py:458`
**Impact:** Internal error messages (file paths, variable values, stack traces) are returned to API clients in JSON responses. Attackers can use these to map the project structure and find injection points.

### 🟡 HIGH-6: Test assertion weakness — dominant pattern asserts nothing about correctness
**File:** 50+ locations across `tests/test_core.py`, `tests/test_edge_cases.py`, `tests/test_swift_bridge_contract.py`
**Impact:** The pattern `assert r is not None` (or `assert isinstance(r, dict)`) appears 50+ times and tests only that the function returned *something*. A function returning an empty dict or error dict passes. This gives a false sense of test coverage — a green suite with weak assertions is not meaningful.

### 🟡 HIGH-7: Sleep-based server startup — 8 instances → flaky CI
**File:** `tests/test_license.py` at lines 319, 329, 340, 357, 410, 440, 466, 485
**Impact:** `time.sleep(0.3)` before asserting server readiness is a classic race condition. If the server takes 0.4s to start, tests fail flakily. If it starts in 0.01s, each test wastes 0.29s. Should use a retry loop with backoff.

### 🟡 HIGH-8: Return code `2` instead of `EXIT_ERROR` (`1`)
**File:** `core/cli.py:498,502,648`
**Impact:** `_run_cf_batch` and `_run_batch` use `return 2` on error paths, but `cli.py` defines `EXIT_ERROR = 1`. Any caller checking `rc == EXIT_ERROR` (i.e. `rc == 1`) will miss these failures. Any sanity/shell automation checking `rc != 0` works, but the contract is broken.

## 4. 🟢 Medium Issues (9)

### 🟢 MEDIUM-1: TOCTOU race condition in license test server port selection
**File:** `tests/test_license.py:296-303`
**Impact:** Opens a socket on port 0 (ephemeral), gets the port, closes the socket, then starts the server on that port. Between close and start, another process can steal the port.

### 🟢 MEDIUM-2: Malformed BOM test — tests garbage characters, not real BOM
**File:** `tests/test_edge_cases.py:95`
**Impact:** The literal string `"ï»¿a,b,c..."` in source is NOT the UTF-8 BOM character `\ufeff` — it's the UTF-8 byte sequence rendered as Latin-1 characters. When the source file is saved as UTF-8 (which it is), these three chars are `Ã¯Â»Â¿`, not the BOM byte-order mark. The test is testing garbage input, not a real BOM.

### 🟢 MEDIUM-3: `test_json_to_toml_large` references non-existent `output_size` key
**Files:** `tests/test_edge_cases.py:452,498`
**Impact:** Asserts `r["output_size"] > 10000` but `convert()` in `configforge.py` does not return an `output_size` key. Returns `KeyError` at runtime. This test must be failing or was never run. Since the test suite shows 0 failures, this suggests either the test is not executing or the return format includes `output_size` somewhere unexpected.

### 🟢 MEDIUM-4: Comment sentinel key collision in JSON round-trip
**File:** `core/configforge.py:1676-1706`
**Impact:** Uses `__cf_comments__`, `__cf_data__`, `__cf_blanks__` as metadata carrier keys for JSON comment preservation. If a real JSON document legitimately uses any of these keys, they are silently consumed and stripped from the output.

### 🟢 MEDIUM-5: Negative expiry timestamps silently ignored
**File:** `web/license.py:167`
**Impact:** If `x` (expiry) is set to a negative number (e.g., `-1`), `meta.get("x")` is truthy, then `meta["x"] > 0` is `False`, so the expiry check is skipped. A negative timestamp should be treated as expired.

### 🟢 MEDIUM-6: Hardcoded magic numbers in health endpoint
**File:** `web/api.py:49-50`, `tests/test_api.py:79`
**Impact:** `MODELS_COUNT = 7` and `TESTS_PASSING = 771` are hardcoded constants that will inevitably drift from actual project state, making `/health` responses permanently inaccurate and the corresponding test brittle.

### 🟢 MEDIUM-7: Naive datetime + "Z" suffix
**File:** `core/models.py:30`, `core/tools.py:434`
**Impact:** `datetime.utcnow()` returns a timezone-unaware datetime, then `.isoformat() + "Z"` appends "Z". This is technically incorrect per ISO 8601 — a naive datetime should not have a timezone suffix. Should use `datetime.now(timezone.utc)`.

### 🟢 MEDIUM-8: No request body size limits — OOM vector on all three servers
**Files:** `web/api.py:162`, `web/license_server.py:73`, `web/serve.py:354`
**Impact:** All three HTTP servers read `Content-Length` bytes into memory without any cap. An attacker can trivially OOM the server by sending a 2GB Content-Length header.

### 🟢 MEDIUM-9: Non-deterministic test assertion — "contradictory" expected behavior
**File:** `tests/test_edge_cases.py:176`
**Impact:** `assert data == [] or data == [{}]` — passes regardless of which of two contradictory behaviors the code exhibits. Masks real bugs.
